0. Process-level period extraction + G5 phase classifier wiring
---------------------------------------------------------------
Status: spec exists (notes/phase_detector.txt §G5, also written up in
notes/presentation_stack_v2_en.html), implementation does NOT.

What is missing:
  FISH currently classifies SHAPES (per callback-invocation Typed DAG)
  via shape_phase_classifier.py (added 2026-04-27). That is
  callback-level. Orthogonal to it, the original G5 spec is
  PROCESS-level: detect period boundaries (cudaStreamSynchronize +
  >5ms idle gap, OR per-stream gap-based partition v2), then label
  each period INIT / WARMUP / STEADY / TAIL / COALESCED(N) / UNKNOWN
  using init_score over the period's API-name list and structural
  signature matching.

Why we need it now:
  Callback-shape classification cannot detect the real INIT phase
  because cuInit / cuLibraryLoadData / cudaMalloc bursts happen in
  the process startup period BEFORE any callback fires. Without the
  process-level period extractor we cannot:
    - identify the INIT period (or WARMUP) at all,
    - segment the trace into frames so per-frame statistics
      (response time, end-to-end latency, jitter) become measurable,
    - tie process-level phase to callback-level shape (e.g. "is the
      anomaly shape only seen during WARMUP periods?").

Concrete plan:
  - Implement period_detector.py in postprocess/:
      input  = nsys sqlite per pid + (optional) ros2_trace
      output = list[(start_ns, end_ns, api_names_within, dag_within)]
  - Wire init_score and the 6-rule classifier (G5 step 3) onto each
    period.
  - Persist per-period labels onto an EX-level attribute
    (EX.A_v["periods"]) similar to what we did for gpu_task_graph.
  - Cross-reference: for each callback shape, record the SET of
    process-level period labels it was observed in. If a shape only
    appears in INIT periods it gets the callback-level "init"
    treatment; otherwise the existing shape-level rules apply.
  - Update shape_phase_classifier.py to consume EX.A_v["periods"]
    instead of trying to score init at shape level.

Open questions (parked):
  - Period boundary heuristic: sync+gap is brittle for streams with
    no host syncs (regime B). v2 spec mentions "GPU-side gap-based
    partition per stream" — needs more precise definition.
  - For multi-stream workloads where streams progress independently,
    do we define one period (taking the union of stream activity) or
    per-stream periods?

Discussed: 2026-04-27, after callback-level classifier showed init=0
on every callback (mechanical reason: no init tokens in any callback
DAG). This task is PRE-condition for completing #37
(formalize phase detection, paper-ready).


0c. Replace babeltrace2 stdout pipe with Python bindings / SQLite sink
----------------------------------------------------------------------
Observed 2026-04-19 on Autoware trace (fish_20260419_211621):
  ingest_ros2_trace streamed bt2 stdout into Python, collected 16M
  events. On noisy traces bt2 writes enough to stderr (176 tracer-
  discarded warnings, 121K bytes in our case) to fill the 64 KiB
  stderr pipe buffer. Since we only drained stdout, bt2 then blocked
  in pipe_write(stderr), Python blocked in pipe_read(stdout), and the
  pipeline deadlocked at ~3.29M committed events.

  Patched (see ingest.py change 2026-04-19): redirect stderr to a log
  file so it can't fill. This is a band-aid — the pipe architecture is
  still fragile (any future stderr growth, any blocking MongoDB call,
  any slow disk, and throughput drops).

Better approach:
  babeltrace2 ships Python bindings (`bt2` module) that expose Message
  Iterators directly — no subprocess, no pipe, no text parsing. Each
  event is a structured Python object with fields we can persist
  straight into MongoDB / a SQLite cache / a parquet file.

  Alternatives we considered:
    (a) bt2 Python bindings → MongoDB directly. Cleanest. Event objects
        are already typed; we skip _parse_trace_line entirely.
    (b) Custom bt2 sink plugin (C++ or Python) that writes SQLite. Then
        ingest reads SQLite. Separates trace-decode from DB-write — a
        potential checkpoint/resume boundary for very long traces.
    (c) Use sqlite-only from the start via a custom LTTng relayd sink
        → likely overkill for now.

  Recommendation: implement (a) first (smallest change), measure
  throughput vs current pipe approach. If throughput beats 28K docs/s
  and we want a reusable intermediate format, add (b) as an option.

Open questions:
  - Does the bt2 Python binding handle all our tracepoint payload types
    (strings, arrays, enum) correctly? Need to confirm with
    ros2:fish_rclcpp_publish_link + rclpy:* events specifically.
  - Memory footprint of the bt2 message iterator on a 16M-event trace.

0e. Ingest nsys SCHED_EVENTS + SAMPLING_CALLCHAINS
--------------------------------------------------
Two large nsys tables we deferred while ingesting the smaller ones
(2026-04-19). Both are time-series-adjacent and need their own
handling pattern.

SCHED_EVENTS (1.85M rows in the aircraft nsys):
  cols: start, cpu, isSchedIn, globalTid, threadState, threadBlock
  What: OS scheduler events — thread context-switched in/out, which
        CPU core, what state (running/runnable/blocked/...) and what
        blocked it. Essentially Linux `perf sched` data captured
        via nsys.
  Value: CPU-side response-time analysis absolutely needs this.
        sched_migrate_task equivalents + per-thread CPU residency +
        block-reason classification (I/O, mutex, lock).
  Plan: InfluxDB measurement `cpu_sched` with tags
        (session, container, cpu, globalTid, threadState,
         threadBlock) and fields (isSchedIn). ENUM_SAMPLING_THREAD_STATE
        and ENUM_SCHEDULING_THREAD_BLOCK resolve the state + block
        codes.
  Scope: large. One aircraft trace is 1.85M rows, a busy Autoware run
        could be 10-100x that. Consider optional ingest (flag) or
        downsampling for very large sessions.

SAMPLING_CALLCHAINS (2.28M rows):
  cols: id, symbol, module, kernelMode, thumbCode, unresolved,
        specialEntry, originalIP, unwindMethod, stackDepth
  What: CPU periodic sampling — every X ms nsys snapshots the call
        stack of running threads.
  Parent table: we need to locate the "SAMPLES" / "COMPOSITE_EVENTS" /
        analog that gives each `id` a timestamp + globalTid. Possibly
        in a different table name in nsys 2025.x. Needs investigation.
  Value: per-callback "what fraction of CPU time in onnxruntime conv /
        do_yolo / ..." — direct input to per-callback roofline and
        CPU-bound vs memory-bound classification.
  Plan (after parent table located): store as `cpu_samples` with
        (session, container, globalTid, stackDepth) tags and
        (symbol, module, originalIP) fields, time-aligned via parent
        sample's start.

Both are BLOCKING for future CPU-side analysis (per-callback PMU
attribution, CPU-bound vs memory-bound classification) but not for
immediate GPU-side DAG + roofline work. Keep on this list.

5. Trace data volume vs accuracy tradeoff model
-------------------------------------------------
Problem:
  Current trace data volume is extremely large. A ~30 second simulation
  session produced:
    - ros2_trace: 8,628,771 events
    - gpu_kernels: 817,320 rows
    - cuda_runtime: 984,338 rows
    - nvtx_events: 214,208 rows
    - gpu_memcpy: 75,145 rows
    - Total ingest time: ~12 minutes

  This does not scale to longer sessions (minutes to hours). Both nsys
  and ros2_trace need an optimal data volume vs accuracy tradeoff.

  nsys side:
    nsys supports sampling frequency configuration. Lower frequency
    reduces data volume but may miss short-lived GPU kernels or bursty
    CUDA API patterns. We need a statistical model that, given a chosen
    sampling frequency, provides a confidence interval for the accuracy
    of derived metrics (kernel duration distributions, API call rates,
    memory bandwidth utilization).

    Approach:
      1. Use current full-rate trace as ground truth
      2. Subsample at various frequencies (1/2, 1/5, 1/10, 1/50, 1/100)
      3. For each subsample, compute key metrics (mean/p50/p99 kernel
         duration, API call rate, memory throughput)
      4. Compare against ground truth, derive error bounds
      5. Fit a model: f(sampling_frequency) → confidence_interval(metric)
      6. Recommend a default frequency that keeps error < X% at 95% CI

  ros2_trace side:
    LTTng captures every single event (publish, take, callback_start/end).
    For systems with high-frequency topics (e.g., IMU at 200Hz), this
    generates massive volumes. Options:
      - Filter by topic/event type at capture time (lttng enable-event)
      - Post-capture downsampling for rate-based analysis
      - Windowed capture (trace 5s every 30s) for steady-state profiling
    Need to quantify: how many seconds of full-rate trace are needed to
    accurately characterize callback latency distributions, scheduling
    jitter, and message flow rates?

  Deliverable:
    A script or notebook that takes a full-rate trace, performs the
    subsampling analysis, and outputs a recommendation table:
      | Sampling config | Data reduction | Metric accuracy (95% CI) |

Priority: HIGH (blocks scaling to real mission durations and multi-drone
  sessions — current 30s trace already takes 12min to ingest)

8. Service client continuation function (split callback model)
---------------------------------------------------------------
Problem:
  When an entity has a client aspect (calls a service), its callback
  execution is split into two phases:

    Phase 1: entity callback runs, sends request (cli aspect)
    Phase 2: response arrives, handle_response runs (continuation)

  These are two separate executor dispatch cycles. Currently the model
  has a single function per entity — the response dep edge returns to
  the SAME function. This is incorrect: the continuation should be a
  separate function vertex.

  Correct model:
    [F] entity_cb ──request──> [F] server_cb
    [F] server_cb ──response──> [F] entity_cb_continuation

  Current model (wrong):
    [F] entity_cb ──request──> [F] server_cb
    [F] server_cb ──response──> [F] entity_cb  ← same function

Implementation:
  1. For each entity with cli aspect, create a second function vertex
     (continuation) linked to fish_rclcpp_client_response_received trace event
  2. Response dep edge targets the continuation function, not the original
  3. Continuation function gets callback_start/end timing from
     handle_response (when callback variant is used)

  Requires: fish_rclcpp_client_response_received tracepoint firing for
  internal node-to-node service calls (currently only fires for
  rclpy ActionClient CLI calls, not rclcpp internal clients).

  First real test case: Autoware on Jazzy — expected to have internal
  rclcpp node-to-node service calls where both client and server are
  persistent traced nodes.

  Dependencies:
    - Jazzy port (jazzy_port.txt)
    - Autoware trace session with internal service calls

Priority: HIGH (after Jazzy port — first thing to verify on Autoware run)


9. Per-callback roofline model via hardware performance counters
-----------------------------------------------------------------
Problem:
  FISH captures callback execution time (callback_start/end) but not
  computational characteristics. We don't know if a callback is
  compute-bound, memory-bound, or cache-contention-limited.

  A cache-aware roofline model per callback would enable:
    - Identify compute vs memory bound callbacks
    - Estimate shared bus contention between concurrent callbacks
    - Cache-related delay analysis (L1/L2/L3/DRAM hierarchy)
    - Data transaction volume per callback
    - Input for schedulability analysis (WCET decomposition)

Approach:
  Use Linux perf PMU counters system-wide alongside FISH tracing.
  PMU monitoring has a fixed, system-independent overhead (HW register
  reads, context switch save/restore). No per-process attach needed.

  Data collection (two concurrent streams):

    A. Per-CPU PMU counters:
       perf stat -a -A -e cycles,instructions,cache-misses,cache-references,\
         LLC-load-misses,LLC-store-misses,bus-cycles -I 1 \
         -o $SESSION_DIR/perf_counters.csv

       The -A flag reports per-CPU counters (not aggregated).
       Run concurrently with LTTng trace session. Start in trace_session.sh
       start_session(), stop in stop_session().

    B. CPU migration tracking:
       LTTng kernel tracepoint: sched_migrate_task
       Fires on every thread migration: (pid, old_cpu, new_cpu, timestamp)
       Much lower volume than sched_switch (only on migration, not every
       context switch).

       Requires: lttng-modules installed, root access, kernel tracing enabled.
       Add to fish_events.txt as kernel event:
         lttng enable-event -k sched_migrate_task

       Alternative (lower fidelity): use cpu_id field already present in
       every LTTng UST event — shows CPU at tracepoint fire time but misses
       inter-tracepoint migrations.

  Analysis (per callback, migration-aware):
    1. callback_start/end gives time window [t1, t2] and initial cpu_id
    2. sched_migrate_task events within [t1, t2] for this vtid give
       the CPU timeline: [(t1, cpu3), (t_mig, cpu7), ..., (t2, cpu7)]
    3. For each (cpu, time_slice) segment:
       - Read per-CPU PMU counter delta from perf_counters.csv
       - Attribute that delta to this callback
    4. Sum across all CPU segments → callback's total PMU values
    5. Derive per-callback metrics:
       - IPC = instructions / cycles
       - Cache miss rate = cache-misses / cache-references
       - LLC miss rate = LLC-load-misses / (LLC-load-misses + LLC-load-hits)
       - Operational intensity = instructions / (cache-misses * cache_line_size)
    6. Plot on cache-aware roofline

  Why migration tracking is critical:
    Without it, per-CPU counter attribution is incorrect:
      - Thread migrates cpu3→cpu7 mid-callback
      - cpu3's counter delta includes OTHER threads' work after migration
      - cpu7's counter delta includes OTHER threads' work before migration
      - Naive attribution double-counts or misattributes

    With sched_migrate_task:
      - Exact time slices per CPU known
      - Counter deltas split proportionally
      - Precise per-callback roofline datapoint

  GPU callbacks (via nsys):
    nsys already provides kernel-level compute and memory throughput.
    Can generate GPU roofline datapoints directly from:
      - Kernel duration + SM occupancy → achieved compute
      - memcpy bytes / duration → achieved bandwidth
      - Register usage → occupancy limit

  P-MoVE/SuperTwin integration:
    PMU abstraction layer from P-MoVE handles Intel/AMD/ARM event name
    differences. CARM model construction reuses P-MoVE's microbenchmark
    approach for establishing cache-level bandwidth ceilings.
    See reference_supertwin_pmove.md for details.

  Deliverable:
    - perf + sched_migrate_task integration in FISH trace session
    - Per-callback migration-aware PMU attribution
    - Roofline plot generation (cache-aware, separate CPU and GPU)
    - Reference: https://sips.inesc-id.pt/~ilic/roofline.php

Priority: HIGH (enables precise per-callback compute/memory characterization,
  shared bus contention analysis, and WCET decomposition — strong paper
  contribution building on SuperTwin/P-MoVE heritage)


10. Multi-container trace composition for unified task model
--------------------------------------------------------------
Problem:
  In multi-container deployments like AAS, each container runs a subset
  of the robotic system (aircraft, ground station, simulation). FISH
  currently traces a single container. The complete task model requires
  composing traces from multiple containers into a single graph.

  Example (AAS):
    - aircraft container: px4_interface, px4_offboard, yolo, kiss_icp
    - simulation container: ros_gz_bridge, PX4 SITL, Gazebo
    - ground container: ground station nodes

  Currently, topics crossing container boundaries appear as external
  entities (EXT). With multi-container composition, these EXT entities
  would be replaced by real entities from the other container's trace.

Approach:
  1. Install FISH in each container independently (setup_fish.sh)
  2. Run trace session in each container concurrently (synchronized
     via /clock topic or NTP)
  3. After collection, compose traces:
     a. Ingest each container's trace into separate MongoDB databases
     b. Extract model per container (identify_executors → entities → functions)
     c. Merge models: match EXT entities across containers by topic/service name
        - EXT publisher in container A matches real subscriber in container B
        - EXT subscriber in container A matches real publisher in container B
        - Replace EXT→real edges with cross-container real→real edges
     d. Unified graph: all executors from all containers, cross-container
        edges marked with container origin and network latency

  Timestamp alignment:
    - All containers share /clock topic (simulated time) — timestamps align
    - For real deployment: NTP sync + clock offset correction
    - Cross-container message latency measurable from publish→take timestamps

  Visualization:
    - D3 interactive: color-code nodes by container origin
    - Grafana: per-container dashboards + unified cross-container view

  Deliverable:
    - fish_compose.py: takes N session directories, merges into unified model
    - Cross-container edge detection via topic/service name matching
    - Unified fish_graph.json for D3 visualization

Priority: HIGH (required for complete system model in multi-container
  deployments — paper contribution: "FISH composes distributed traces
  into a unified scheduling hierarchy")


11. Log active settings at daemon startup
------------------------------------------
Problem:
  The daemon does not log which settings were active when it started.
  fish_events only records stabilization parameters (cl_interval,
  cl_count, ex_interval, ex_count). Other important settings like
  replay_rosbag, replay command, settle_time, poll_interval, nsys flags,
  and SKIP_PATTERNS are not recorded anywhere in the session output.

  This makes it difficult to diagnose issues after the fact — e.g.
  "was replay_rosbag true or false in this session?" requires checking
  the container's fish_settings.ini, which may have changed since.

Fix:
  At daemon startup (after "started" log entry), dump all active settings
  to fishlog as a single multi-line event or a separate settings snapshot
  file. Two options:

  A. fishlog event (compact):
     <ts> daemon - - fish_daemon settings replay_rosbag=false replay_cmd= settle_time=3.0 poll_interval=1.0 ...

  B. Separate file (complete):
     Write fish_settings.ini contents to <session>/fishlog/active_settings.ini
     This preserves the exact config used, including comments.

  Option B is simpler and more useful for reproducibility.

Priority: LOW (quality of life, not blocking functionality)


13. Invariant validation on graph mutations
---------------------------------------------
Problem:
  graph_store.py exposes mutation primitives (update_node/edge, add_node/edge,
  remove_node/edge) but does not validate that resulting graphs still honor
  the FISH hierarchy invariants:

    CN (L-1) → EX (L0) → N (L1) → E (L2) → F (L3)

  A careless mutation can leave the graph in an inconsistent state:
    - A node with non-monotonic level vs. parent
    - An EX with no CN parent
    - Communication edges between types that should never communicate
    - children[] pointing at missing / wrong-type IDs
    - External vertices without the "external" flag set

Why deferred:
  Implementing validators right now complicates testing of the round-trip
  (save → load → viz). We want to iterate on the storage layer first and
  only bolt on invariants once the happy path is solid.

Implementation:
  1. Define INVARIANTS in a single place (validators.py):
       - type_to_level: {"CN": -1, "EX": 0, "N": 1, "E": 2, "F": 3}
       - allowed_contain_edges: {(-1, 0), (0, 1), (1, 2), (2, 3)}
       - allowed_comm_edges:    {(F, E), (E, F), ...} from existing model
       - every node has a parent via "children" except CN
       - external flag consistent with whether the node has outside edges
  2. graph_store.validate(session_name) → list of issues
  3. Mutation methods accept validate=True flag (default False initially) —
     raise ValidationError on violation, rollback.
  4. CLI: `python graph_store.py validate <session>`
  5. Add tests/invariants.py with unit tests: known-good graph passes,
     each expected violation is caught.

Priority: MEDIUM (not blocking storage layer work; add once roofline /
  other analyses start producing non-trivial mutations that could break
  invariants silently).


14. Evaluate alternative graph layout engines for fish_viz
-----------------------------------------------------------
Problem:
  D3 force layout struggles with large graphs (3000+ nodes). The current
  tree-stair layout with hierarchical expansion (G operator) solves the
  node count problem, but the layout itself is simplistic — horizontal
  positioning is just sequential, no edge crossing minimization, no
  grouping by communication density.

Candidates to evaluate:

  A. ELK.js (Eclipse Layout Kernel):
     - Java-based layout engine compiled to JS via GWT
     - Excellent hierarchical/layered layout (Sugiyama-style)
     - Built-in support for compound nodes (parent-child grouping)
     - Edge crossing minimization, port-based routing
     - Used by Eclipse IDE, JetBrains, and many EDA tools
     - npm: elkjs
     - Would replace the stair layout computation, keep D3 for rendering

  B. Cytoscape.js:
     - Full graph visualization library (not just layout)
     - Native compound node support (parent contains children visually)
     - Multiple layout algorithms: dagre, klay (ELK-based), cose-bilkent
     - Better performance than D3 for large graphs
     - Would replace both layout AND rendering (D3 no longer needed)
     - Has built-in expand/collapse for compound nodes

  C. Dagre (dagre-d3):
     - Lightweight directed graph layout
     - Sugiyama-style layered layout with edge routing
     - Works with D3 for rendering
     - Simpler than ELK but less features

  Evaluation criteria:
    - Layout quality for hierarchical ROS 2 graphs
    - Support for the G expansion operator (compound nodes)
    - Edge routing clarity (orthogonal, crossing minimization)
    - Performance with 300-3000 nodes
    - Integration effort with existing fish_viz.html

  Approach: build a small prototype with each, compare on the same
  fish_graph.json. Pick the best one and rewrite fish_viz.html.

Priority: MEDIUM (current stair layout works, but better layout would
  significantly improve readability for paper figures)


15. Test strategy + GitHub Actions CI/CD (release blocker)
-----------------------------------------------------------
Date added: 2026-05-17
Context: pivot to release-ready software. A research tool of this size
(~10-20 KLOC Python, single-author, transitioning to release) does
NOT benefit from broad unit-test coverage — the bugs that matter live
at boundaries (trace schema drift, paper-number drift after refactor,
session-filter forgotten somewhere on the shared `fish` InfluxDB).
Wrong shape: 90% line coverage on every helper. Right shape: a few
"regression anchor" tests that guard the load-bearing claims.

Test pyramid (inverted from typical):

  Tier 1 — Pure-logic unit tests  (tests/unit/, fast, no DB)
    - stream-bridge attribution Rule A/B/C
    - LCS subseq vs LCS substring
    - trie conditional probability calc
    - phase classifier rule scoring (init_score, anomaly tests)
    - callback split logic
    - shape_signature determinism
    - kernel-name shortener
  These are the ONLY things worth typical unit tests — pure functions,
  no external state, refactor-prone.

  Tier 2 — Fixture contract tests  (tests/contract/, ~2 min with services)
    Fixtures in tests/fixtures/mini_session/ (< 50 MB, repo-committed):
      a 30 s standalone trace with CTF + nsys.sqlite + snapshot dir.
    Tests:
    - babeltrace2 line parser yields expected dict for known input
    - nsys CUPTI_ACTIVITY_KIND_KERNEL / RUNTIME / MEMCPY rows carry
      the fields ingest expects
    - InfluxDB filter regression guard:
        unfiltered SELECT yields rows from > 1 session
        filtered SELECT yields rows from exactly that (session, container)
    - ScopedMongo correctly suffixes scoped collection names
    - graph_store save_graph → load_graph round-trip preserves
      vertex attrs (t_v, id_v, A_v, level)

  Tier 3 — One golden-session smoke  (tests/e2e/, ~5-10 min, nightly+pre-release)
    Bigger fixture (or download in CI). Full pipeline:
      ingest_session → identify_executors → identify_entities →
      identify_callbacks → attribute_aspects → attach_callback_groups →
      detect_oort_threads → split_callbacks → create_graph →
      gpu_dag.extract → save_graph
    Assertions against tests/paper_anchors.json — a single source of
    truth listing load-bearing numbers (≈ 8-10 entries):
      fish_20260427_165116:
        n_kernels: 14576
        n_runtime: 85263
        graph_nodes: 3160
        graph_edges: 5874
        oort_threads: 56
      fish_compose_20260419_161633 / aircraft:
        n_kernels: 854495
        n_runtime: 1029170
        F1696_invocations: 45
        F1696_distinct_shapes: 25
      (etc.)
    Drift > some tolerance → CI red → either fix the code or update
    the paper.

GitHub Actions integration:

  .github/workflows/ci.yml (push + pull_request):

    jobs:
      lint:
        runs-on: ubuntu-latest
        steps: ruff/black/mypy --strict postprocess/

      unit-contract:
        runs-on: ubuntu-latest
        services:
          mongo:
            image: mongo:7
            ports: ['27017:27017']
          influxdb:
            image: influxdata/influxdb:3-core
            ports: ['8181:8181']
            env: { INFLUXDB3_AUTH_TOKEN: test-token }
        steps:
          - actions/checkout@v4 (lfs: true)
          - actions/setup-python@v5
          - actions/cache for ~/.cache/pip
          - pip install -r requirements.txt -r requirements-dev.txt
          - pytest tests/unit tests/contract -v

      smoke:
        if: github.event_name == 'schedule' || contains(github.ref, 'release-prep')
        runs-on: ubuntu-latest
        services: { mongo: ..., influxdb: ... }
        steps:
          - download golden fixture (LFS or gh release asset)
          - pytest tests/e2e --paper-anchors tests/paper_anchors.json
          - on drift: open GH issue with diff report

  .github/workflows/release.yml (on: push: tags: v*):
    - build wheel (python -m build)
    - upload to GH Release attachments
    - (later) PyPI via twine

  .github/dependabot.yml — weekly Python deps PRs

  Branch protection on main: require ci.yml green; force-push off;
  signed commits optional.

Fixture storage decision tree:
  < 50 MB     → commit directly to repo (tests/fixtures/mini_session/)
  50 MB - 1 GB → Git LFS (free 1 GB storage / month)
  > 1 GB      → upload as GH Release asset, gh release download in CI

CUDA/ROS NOT needed in CI: pre-recorded CTF + pre-recorded nsys.sqlite
mean no NVIDIA runner required. babeltrace2 installs via apt.

Implementation order:
  Step 1. Record a fixture mini_session (~30 s standalone trace, < 50 MB).
          Without this every other test is paper-mâché.
  Step 2. tests/paper_anchors.json — list the 8-10 load-bearing numbers
          that paper currently cites (audit paper_important_points.txt
          and fish_computable_properties.txt for what is quoted).
  Step 3. Tier 1 unit tests (pure logic) — cheapest to write, no infra.
  Step 4. Tier 2 contract tests + mini_session fixture.
  Step 5. ci.yml with lint + unit-contract jobs only. Branch protection on.
  Step 6. Golden-session smoke + smoke job + paper_anchors comparison.
          Nightly cron + release-prep branch trigger.

What NOT to do:
  - Coverage gate (% lines) — wrong metric, encourages noise tests.
  - Mock the database in integration tests — past lesson: real DB
    catches schema drift; mocks pass while prod breaks.
  - Test every helper. Pure logic at boundaries only.

Effort estimate:
  lint + unit + contract job working in CI: 1 day
  mini_session fixture + paper_anchors.json + tier 2: 2-3 days
  golden-session smoke + drift report job: 2 days
  Total: ~1 week for full pipeline.

Priority: HIGH (release blocker — without anchor tests, future
  refactors silently invalidate paper numbers; without CI, contributors
  cannot land changes safely).
