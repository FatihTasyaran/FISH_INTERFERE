2. Why rclpy Tracepoints Don't Fire
-------------------------------------

Problem: ros2 action send_goal CLI uses rclpy's ActionClient class, which
does NOT use rclpy.client.Client or the executor's _take_client/_execute_client.

  ActionClient (rclpy/action/client.py):
    - Inherits from Waitable, not Client
    - Has its own take_data() and execute() methods
    - Calls _client_handle.send_goal_request() directly (C pybind11)
    - Response handling in execute() via take_goal_response/take_result_response
    - Completely bypasses client.py and executors._execute_client

  Our patches are in:
    client.py:call_async()       → ActionClient doesn't call this
    executors.py:_execute_client → ActionClient doesn't go through this

  This is analogous to the rclcpp action server situation:
    - Action servers use Waitable interface, bypass ServiceBase
    - Action clients use Waitable interface, bypass ClientBase

5. Relationship to Entity Model
---------------------------------

Per FISH design (comm_dep_models.txt):
  - Client is an ASPECT, not an entity type
  - Entity types = callback types = sub, serv, tmr
  - An entity with client aspect calls a service inside its callback
  - The request_sent event fires inside the entity's callback window
  - The response_received event fires later in a separate executor dispatch

The tracepoints enable:
  - Round-trip latency measurement (request_sent.ts → response_received.ts)
  - Sequence number matching (request ↔ response)
  - Response type identification (future-only vs callback)
  - Attribution to the requesting entity via callback window matching
