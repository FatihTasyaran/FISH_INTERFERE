FISH GPU task-graph extraction — 4-layer stack

Input: CUPTI tables (RUNTIME, KERNEL, MEMCPY, MEMSET, SYNCHRONIZATION) + ros2_trace callback_start/end
▼
1. Period detection — "Where do DAG invocations begin and end?"
sync-boundary pattern: cudaStreamSynchronize followed by > 5 ms idle gap marks boundary. Alternative (v2): GPU-side gap-based partition per stream (data available, integration pending).
▼
list of (start_idx, end_idx) pairs per tid / stream
▼
Typed-DAG builder: for each period, emit CPU/SM/CE nodes + launch, stream_fifo, cpu_order edges.
▼
list of Typed DAGs (one per period)
▼
2. Shape detection — "Which periods are structurally equivalent?"
signature hash over (pe, name) sequence + sorted edges. Exact match only. Future: edit-distance clustering (embed_start.txt).
▼
{signature → count + template DAG}
▼
3. Phase detection — "What lifecycle role does each period play?"
init_score weighted by CUDA-init API tokens + rule classifier. Labels: INIT / WARMUP / STEADY / TAIL / COALESCED(N) / UNKNOWN.
▼
labelled periods + primary phase per shape
▼
4. Stream-bridge attribution — "Which callback invocation owns each CUDA call?"
correlationId → streamId join. Ownership rules:
(A) same-tid time-window membership
(B) stream-claim bridge (cross-tid dispatch)
(C) oort residual
▼
Output: per-callback gpu_dags + inter-stream wait-event graph (node counts deterministic from CUPTI; edges deterministic when --cuda-event-trace is enabled, otherwise time-proximity heuristic)

Phase detection — mathematical definition

Setup

Let P = (p1, …, pn) denote the chronologically ordered set of periods on one tid/stream.
For each pi, let:
   A(pi) = sequence of CUDA API names in the period
   D(pi) = (Vi, Ei) the Typed DAG of the period, with Vi = ⟨(pej, namej)⟩
   pos(pi) = i ∈ [1, n]

Init-score σ

Let 𝒯 = {t1, …, tk} be the set of init-signal tokens, each associated with a threshold θt ∈ ℕ ∪ {∞} and a weight wt ∈ ℕ.

ct(p) := |{ a ∈ A(p) : t ⊆ a }|   (count of token occurrences) σ(p) := Σt ∈ 𝒯 𝟙[θt ≠ ∞] · wt · 𝟙[ct(p) ≥ θt] + 𝟙[θt = ∞] · wt · ct(p)

Shape signature & dominant shape

sig(p) := SHA1( ⟨(pej, namej)⟩j ∥ sorted(Ep) )[0..12] 𝒫prod := { p ∈ P : σ(p) < 20 } sig* := argmaxs |{ p ∈ 𝒫prod : sig(p) = s }| seq* := node sequence of the canonical DAG D(p) for any p with sig(p) = sig*

Label assignment

L(pi) = INIT             if  σ(pi) ≥ 100 L(pi) = STEADY           else if  sig(pi) = sig* L(pi) = COALESCED(N)   else if  ∃ N ∈ ℤ≥2 : |seq(pi)| = N · |seq*| and ∀k, seq(pi)[k] = seq*[k mod |seq*|] L(pi) = TAIL             else if  pos(pi) ≥ n − 3 ∧ 10 ≤ |seq(pi)| < |seq*|                               ∧ ∃m : seq(pi)[0..m] = seq*[0..m], m/|seq(pi)| ≥ 0.95                                 ∧ ∀k ∈ [m, |seq(pi)|) : "Synchronize" ∈ seq(pi)[k].name L(pi) = WARMUP           else if  20 ≤ σ(pi) < 100 ∧ pos(pi) < 3 L(pi) = UNKNOWN          otherwise

Stream-dependency edges — formal definition

Setup

Let 𝒢 = 𝒦 ∪ ℳmemcpy ∪ ℳmemset denote the union of GPU-side activities from CUPTI_ACTIVITY_KIND_KERNEL, MEMCPY, and MEMSET, with each g ∈ 𝒢 carrying g.start, g.end, g.streamId.

Let 𝒮wait = { r ∈ CUPTI_SYNCHRONIZATION : r.syncType = 2 } (StreamWaitEvent rows).

Deterministic part — wait node counts

Count of times each stream serves as the waiting (destination) stream. These are obtained directly from 𝒮wait without inference:

Wdst(s) := |{ r ∈ 𝒮wait : r.streamId = s }|

Edge extraction — two regimes

Regime I — deterministic (requires --cuda-event-trace=true, ISSUE-003):

Each r ∈ 𝒮wait carries a valid r.eventId. The event was recorded on some stream via cudaEventRecord: src(r) := streamId of the most recent cudaEventRecord(e = r.eventId, stream) call before r.start dst(r) := r.streamId edge(r) := (src(r), dst(r)) — certain

Regime II — time-proximity heuristic (when eventId = −1 in the SYNC row, as in all pre-2026-04-20 traces):

𝒢<r := { g ∈ 𝒢 : g.end < r.start ∧ g.streamId ≠ r.streamId } src̃(r) := argmaxg ∈ 𝒢<r g.end    (the most-recently-ended activity on any other stream) edge(r) := (src̃(r).streamId, r.streamId) — inferred
Caveat on regime II. The event-record call is frequently issued inside closed-source libraries (TensorRT, ORT) that do not surface the record through CUPTI's RUNTIME table. The heuristic above is correct when (i) library workers issue their record immediately after a preceding kernel, and (ii) only one other stream was recently active. On dense pipelines (Autoware pointcloud), the heuristic resolves 99.9% of waits. On sparse pipelines (yolo main-loop), 53% — the remainder is left unresolved rather than guessed.

Weighted edge set

For each distinct ordered pair (sa, sb), the edge weight is: Wedge(sa → sb) := |{ r ∈ 𝒮wait : edge(r) = (sa, sb) }|

Sanity check (deterministic)

∀ s : Σa Wedge(sa → s) + (unresolved waits on s) = Wdst(s)
