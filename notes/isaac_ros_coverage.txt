Isaac ROS coverage tracker — FISH validation roadmap
====================================================

Date created: 2026-05-17
Source: https://nvidia-isaac-ros.github.io/index.html  (Isaac ROS 4.4)

This file tracks every Isaac ROS package / reference workflow against
the FISH validation plan. It is the master inventory; the README's
"Validated on" table surfaces only the top-priority subset.

Coverage taxonomy
-----------------

  P0 — Paper-required.  GPU-heavy, has an isaac_ros_benchmark script,
       has known rosbag input. These become FISH fixtures and feed
       paper_anchors.json.

  P1 — Supporting.      Useful coverage breadth — multi-node graph
       variants, specialised codecs, encoder/decoder pairs.

  P2 — Larger workflow. SLAM / mapping bundles — bigger setup cost,
       fewer fixtures but valuable end-to-end demonstrations.

  P3 — Reference workflow. Manipulation pipelines (cuMotion, pick &
       place, gear assembly). Heavy host-hardware dependency (Franka,
       UR robots). Stretch goal.

  P4 — Infrastructure.  NITROS, GXF, common, nitros bridge. Used by
       every workload; not a workload itself. FISH automatically sees
       these inside any P0–P3 capture.

  P5 — Skip.            Interfaces-only packages, ROS message-only
       packages, RViz plugins, conversion helpers.

Status values
-------------

  planned         — In the queue, no work started.
  setup-ready     — Dev image pulled, deps verified, ready to write
                    mission YAML.
  yaml-drafted    — examples/<workload>.yaml committed; not yet
                    captured.
  captured        — Trace exists in ~/fish_traces/, ingested into
                    MongoDB + InfluxDB.
  anchored        — tests/paper_anchors.json entry written; reference
                    numbers locked in.
  fixture         — Trimmed trace committed under
                    tests/fixtures/<workload>/ (CI/CD smoke test
                    surface).

Hardware reality check
----------------------

  Local rig: RTX A1000 Laptop (sm_86 Ampere, 4 GB VRAM).
  Useful capacity: ≤ ~3.5 GB for model + activations.
  Implications:
    - FoundationPose, FoundationStereo, SAM2 → may not fit in 4 GB
      at full precision. Need FP16 / INT8 variants, or skip on this
      machine until larger GPU is available.
    - DOPE, AprilTag, ESS-light, Rectify → fit comfortably.

  Driver-side CUDA: 13.0 (NVIDIA-SMI 580.126.09).
  Container runtime CUDA: 12.x (whatever Isaac ROS 4.4 base ships).

Tactical order for the next 10 weeks
====================================

The order minimises image-pull duplication and uses each captured
session to derive multiple fixtures where possible.

  W1: isaac_ros_apriltag           — proves the FISH-on-Isaac-ROS path,
                                     smallest GPU footprint, simplest
                                     mission YAML.
  W2: isaac_ros_image_proc         — rectify_node, single-pass GPU
                                     pipeline. Shares image input with
                                     AprilTag.
  W3: isaac_ros_stereo_image_proc  — adds the stereo dimension.
  W4: isaac_ros_ess (Light)        — first DNN inference under FISH;
                                     small-enough model for A1000.
  W5: isaac_ros_dnn_image_encoder  — feeds W6+.
  W6: isaac_ros_tensor_rt          — TensorRT path (covers DOPE + PS).
  W7: isaac_ros_triton             — Triton path; A/B compare to W6.
  W8: isaac_ros_apriltag_graph     — multi-node graph variant of W1.
  W9: isaac_ros_h264_decoder/encoder — video codec workloads.
  W10: isaac_ros_dope              — full pose-estimation pipeline.

Each week we expect 1 new captured session + 1 fixture committed +
1 paper_anchors.json entry. By W10 the Results section's "per-workload"
table is filled.

Workloads explicitly NOT in the 10-week plan:
  - FoundationPose, FoundationStereo, SAM, SAM2 — VRAM check first.
  - cuMotion + manipulation workflows — different hardware requirement.
  - Visual SLAM / mapping — bigger separately-tracked effort.


Note on r2b dataset
===================

Most P0 workloads above accept the same r2b dataset bags (the
camera-only sequences: r2b_lounge, r2b_galileo, r2b_hideaway, ...).
ros2_benchmark's DataLoaderNode replays the bag at controlled rates.
Stereo workloads need stereo r2b bags (r2b_galileo has the HAWK
stereo pair); pose-estimation needs r2b_hope-style scenes.

One bag download covers many workloads → the NGC download is a
one-shot prerequisite, not per-workload.
