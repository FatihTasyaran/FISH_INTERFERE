EXPECTED vs OBSERVED — ROS 2 scheduling behavior under FISH tracing
=====================================================================

Purpose:
  Record every case where the documented / canonical ROS 2 scheduling
  model diverges from what FISH actually observes at runtime. These are
  the findings that drive the paper: the observed behavior is not
  captured by any existing ROS 2 scheduling model we are aware of, nor
  by the official documentation.

  Each entry below follows the same shape:
    - EXPECTED  — what the docs / tutorials imply
    - OBSERVED  — what FISH measured
    - EVIDENCE  — concrete trace session + numbers
    - CAUSE     — which rclcpp code path produces the observed behavior
    - IMPLICATION — why this matters for response-time analysis

  Evidence references the AAS mission trace collected on 2026-04-18:
    session:    fish_compose_20260418_165627
    containers: aircraft, ground, simulation (AAS single-drone w/ PX4 SITL)
    mission:    takeoff -> orbit -> land, ~95 s of active flight
    tracepoints: 4 new scheduler tracepoints (fish_rclcpp_executor_init,
                 fish_rclcpp_callback_group_init, fish_rclcpp_cbgroup_add,
                 fish_rclcpp_executor_add_cbgroup) installed in rclcpp
                 (callback_group.cpp + executor.cpp + 3 executors/*.cpp)


============================================================================
1. A process can host MULTIPLE executor instances
============================================================================

EXPECTED
--------
The canonical ROS 2 tutorial pattern is:

    int main(...) {
        auto node = std::make_shared<MyNode>();
        rclcpp::spin(node);     // or an explicit Executor
    }

Implicit assumption across docs, design papers, and most analytic
scheduling models for ROS 2:
    1 process = 1 executor
    The executor dispatches all callbacks from all nodes inside the process.

OBSERVED
--------
In the AAS trace, six of the eight FISH-observed C++ processes hosted
MORE than one Executor instance during their lifetime:

  process            primary                         extras
  ------------------ ------------------------------- --------------------
  parameter_bridge   SingleThreaded, 1 CG           (none) × 3 processes
  ground_system      SingleThreaded, 2 CGs          1 × SingleThreaded
  kiss_icp_node      SingleThreaded, 4 CGs          3 × SingleThreaded
  state_sharing_node MultiThreaded (20 thr), 4 CGs  1 × SingleThreaded
  px4_offboard       MultiThreaded (20 thr), 4 CGs  1 × SingleThreaded
  px4_interface      MultiThreaded (20 thr), 6 CGs  1 × SingleThreaded

Total: 9 primary executors + 6 extras = 15 Executor instances across
8 C++ processes (plus 1 rclpy process: yolo_py, whose Python executors
are not yet instrumented).

The "extras" split into two categories — see below.

EVIDENCE
--------
  Trace events:
    aircraft  — 13 fish_rclcpp_executor_init events
    ground    — subset of the 13 (3 parameter_bridges split across roles)
    simulation — 3 fish_rclcpp_executor_init events

  Event payload (raw):
    ros2:fish_rclcpp_executor_init {
      executor_addr: "0x7FFCD1A35B30",
      executor_type: "SingleThreaded" | "MultiThreaded" | "StaticSingleThreaded",
      num_threads:   1 | 20 | ...
    }

CAUSE
-----
Two distinct mechanisms produce multiple executors per process:

  (1) Transient executors created by convenience helpers.

      rclcpp/src/rclcpp/executors.cpp:

        void rclcpp::spin(NodeBaseInterface::SharedPtr node_ptr) {
          rclcpp::ExecutorOptions options;
          options.context = node_ptr->get_context();
          rclcpp::executors::SingleThreadedExecutor exec(options);  // <<
          exec.add_node(node_ptr);
          exec.spin();
          exec.remove_node(node_ptr);
        }

      Same pattern in rclcpp::spin_some() and
      rclcpp::spin_until_future_complete(). Every call allocates a fresh
      SingleThreadedExecutor on the stack, constructs it (→ fires
      fish_rclcpp_executor_init), spins, then destructs it.

      Common callers inside nodes (any of these trigger a new executor):
        - Client<T>::async_send_request(req).get()   (blocking sync)
        - Node::declare_parameter() if a remote param service is queried
        - Custom service wait_for_service() loops
        - Lifecycle / parameter callbacks invoking sub-spins

      In kiss_icp_node, three such extras were observed — almost
      certainly from parameter service queries during startup.

  (2) Deliberate persistent parallel executors.

      An application thread can construct its own Executor and spin it
      alongside the main one. The px4_interface process visibly uses
      this pattern:

        MT executor (20 threads, 6 CGs) — main callback dispatch
        ST executor (1 thread,   1 CG)  — auxiliary work (parameter,
                                           lifecycle, low-rate services)

      ROS 2 documentation DOES mention "you may use multiple executors"
      but treats it as an advanced corner case; no analytic model we are
      aware of captures it.

IMPLICATION
-----------
Response-time analysis conventionally assumes a static task set with a
static scheduler. With multiple executors per process:

  - The "scheduling entity" set is time-varying: transient executors
    come and go during startup or specific mission phases.
  - Callbacks of the SAME node may be dispatched by different executors
    at different points (e.g., a parameter callback handled by a
    sub-spin while the main MT loop is idle).
  - Thread affinity of a callback depends on WHICH executor handled it.
  - Priority inversion is possible: an MT callback waits on a service
    whose reply is dispatched by a sibling ST executor in the same
    process.

FISH's approach: enumerate EVERY Executor instance per process and
attach metadata (executor_type, num_threads, cb_groups). The
graph_store `EX` vertex records `executor_addr`, `executor_type`,
`num_threads`, `cb_groups`, plus `extra_executors` (list) for
coexisting instances. Downstream analysis can filter transient
executors (typically no persistent CGs, short-lived) from persistent
ones (multiple CGs, long-lived).


============================================================================
2. A node can own MULTIPLE callback groups with MIXED types
============================================================================

EXPECTED
--------
Two common simplifications in the literature and tutorials:

  (a) Each node has one callback group — the default MutuallyExclusive.
  (b) A process either uses MutuallyExclusive-only or Reentrant-only
      callback groups, depending on the designer's choice of
      SingleThreaded vs MultiThreaded executor.

Neither simplification survives contact with real applications.

OBSERVED
--------
AAS aircraft node CG counts:

  process            CGs  types
  ------------------ ---  ----------------------------------------
  parameter_bridge   1    MutuallyExclusive (auto)
  ground_system      2    MutuallyExclusive (auto) + MutuallyExclusive (manual)
  kiss_icp_node      4    mixed — each of its 4 executors has 1 CG
  state_sharing_node 4    mix of MutuallyExclusive and Reentrant
  px4_offboard       4    mix of MutuallyExclusive and Reentrant
  px4_interface      6    mix of MutuallyExclusive and Reentrant

Overall distribution (across the unified compose graph):
  14 MutuallyExclusive   +   23 Reentrant    (37 CG-attributed entities)

High-frequency PX4 sensor topics are specifically Reentrant:
  /Drone1/fmu/out/vehicle_global_position → Reentrant CG
  /Drone1/fmu/out/vehicle_local_position  → Reentrant CG
  /Drone1/fmu/out/airspeed_validated      → Reentrant CG
  /Drone1/fmu/out/vehicle_odometry        → Reentrant CG

Application authors deliberately mix CG types to control intra-node
concurrency: sensor ingestion in Reentrant so messages can be
processed concurrently on different threads, state mutations in
MutuallyExclusive to avoid races.

EVIDENCE
--------
  ros2:fish_rclcpp_callback_group_init events per container:
    aircraft     — 24 CG init events
    ground       —  3 CG init events
    simulation   —  3 CG init events

  ros2:fish_rclcpp_cbgroup_add events (entity ↔ CG association):
    aircraft     — 537 add events (entities + waitables)

  The cg_add event uses the RCL handle (obtained via
  `subscription_ptr->get_subscription_handle().get()` etc.) so it
  joins directly with the rcl_*_init events FISH already uses to
  identify entities.

CAUSE
-----
rclcpp API allows a node to create CGs freely:

    node->create_callback_group(CallbackGroupType::MutuallyExclusive);
    node->create_callback_group(CallbackGroupType::Reentrant);

Each entity (subscription, timer, service, client) takes a
`callback_group` via its SubscriptionOptions / TimerOptions /
ServiceOptions. If omitted, the entity falls into the node's default
MutuallyExclusive group.

In AAS, px4_interface's 6 CGs are consistent with a separation like:
  - sensor ingest Reentrant group
  - command publish MutuallyExclusive group
  - parameter/lifecycle MutuallyExclusive group
  - timer MutuallyExclusive group
  - service server MutuallyExclusive group
  - default MutuallyExclusive group (for anything left over)

IMPLICATION
-----------
The scheduling unit is the CALLBACK GROUP, not the node. Response-time
analysis must model:

  - For each MutuallyExclusive CG: callbacks in the group are serialized.
    WCRT analysis treats the group as a single resource with FIFO / prio
    order determined by ROS 2 executor policy.
  - For each Reentrant CG: callbacks can run concurrently across any
    number of executor threads. Response time is bounded only by
    callback WCET and executor thread availability (not by other
    callbacks in the same CG).
  - Callbacks in different CGs run concurrently whenever threads are
    available.

Treating a whole node as one scheduling entity fundamentally misses
this structure and produces either overly pessimistic bounds (if
assumed serialized) or optimistic bounds (if assumed concurrent).


============================================================================
3. MultiThreadedExecutor default thread count = hardware_concurrency()
============================================================================

EXPECTED
--------
MultiThreadedExecutor has a constructor parameter `number_of_threads`.
Documentation example frequently passes an explicit value (often 2 or 4).
Many analytic models treat "number of threads" as an application-level
parameter set by the developer.

OBSERVED
--------
All three MultiThreaded executors in AAS were constructed with
`number_of_threads = 20`.

  px4_interface:       MultiThreaded, 20 threads
  px4_offboard:        MultiThreaded, 20 threads
  state_sharing_node:  MultiThreaded, 20 threads

Hardware on the host running the container: 20 physical cores.
None of the AAS node source code specifies `number_of_threads`
explicitly — it is left at the default (0).

CAUSE
-----
rclcpp/src/rclcpp/executors/multi_threaded_executor.cpp line 37:

    number_of_threads_ = number_of_threads
        ? number_of_threads
        : std::thread::hardware_concurrency();
    if (number_of_threads_ == 0) {
      number_of_threads_ = 1;
    }

When the constructor's `number_of_threads == 0` (its default), rclcpp
queries `std::thread::hardware_concurrency()` and uses whatever the OS
reports. This value is determined at RUNTIME, on the machine hosting
the container, and is INVISIBLE to the application source.

IMPLICATION
-----------
  - A model built offline from source inspection will always be wrong
    about `num_threads` for any executor that does not explicitly pass
    a value.
  - The same application binary has different scheduling capacity on
    different hardware — a 4-core machine gets 4 threads, a 20-core
    machine gets 20 threads. Response-time bounds derived on one
    platform do not transfer.
  - For containerized deployments, thread count is the HOST's thread
    count, not the container's CPU quota. An MT executor may spawn 20
    threads even if `--cpus=2` limits actual parallelism. Under load,
    this translates to thread contention inside the container.

FISH captures `num_threads` as measured at the actual constructor call
(tracepoint fires after the hardware_concurrency resolution), so the
recorded value reflects what the executor will use in practice.


============================================================================
4. Python (rclpy) executors are INVISIBLE to C++ tracepoints
============================================================================

EXPECTED
--------
ROS 2 exposes the "same" executor concept in both rclcpp and rclpy.
Tutorials describe the two interchangeably.

OBSERVED
--------
The yolo_py node in AAS (PID 1249, rclpy-based) produced ZERO
fish_rclcpp_executor_init, fish_rclcpp_callback_group_init, fish_rclcpp_cbgroup_add, or
fish_rclcpp_executor_add_cbgroup events. In the graph_store model, its EX
vertex has:

    executor_type: ?        (unknown)
    num_threads:   ?
    cb_groups:     []

CAUSE
-----
rclpy's Executor, CallbackGroup, SingleThreadedExecutor,
MultiThreadedExecutor, MutuallyExclusiveCallbackGroup, and
ReentrantCallbackGroup are pure Python classes in
`rclpy/executors.py`. They do not call into rclcpp's C++ Executor
hierarchy. Tracepoints placed in rclcpp never fire for rclpy nodes.

FISH already has a ctypes bridge pattern (see
`python/rclpy/fish_rclpy_trace.py`) that Python code uses to call
tracetools' C functions. The rclpy scheduler integration is a
mechanically straightforward extension of that bridge: patch
`rclpy/executors.py` __init__ methods to call
`ros_trace_fish_rclcpp_executor_init(...)` etc.

IMPLICATION
-----------
Until the rclpy patch lands, any analysis over a mixed C++/Python
deployment will have "holes" at Python nodes:
  - executor type unknown
  - num_threads unknown
  - CG membership unknown

Also worth noting: rclpy's executors run under the Python GIL, so
MultiThreadedExecutor in rclpy does not buy the same parallelism as in
rclcpp. The eventual rclpy scheduler tracepoints should be reported
with a "python" flag so response-time analysis knows to apply GIL
constraints.


============================================================================
5. fish_rclcpp_callback_group_init fires BEFORE the owning node is known
============================================================================

EXPECTED
--------
A callback group belongs to a node — `node->create_callback_group(...)`
is the common API. Any representation of CGs should naturally carry
the owning node as metadata.

OBSERVED
--------
The fish_rclcpp_callback_group_init event payload has:
    group_addr, group_type, automatic_add, vpid (process id)
but NOT the node_addr.

The node association is recoverable only after the first
fish_rclcpp_cbgroup_add or fish_rclcpp_executor_add_cbgroup that mentions the same
group_addr — at that point we can triangulate:
    E.entity_addr → parent N → node_handle
or
    fish_rclcpp_executor_add_cbgroup(executor_addr, group_addr, node_addr)

In the current FISH extraction this triangulation works: every CG that
ends up in the graph has at least one entity added to it, so we
recover (CG → node) via (CG → entity → node).

CAUSE
-----
The CallbackGroup constructor does not know which node is creating it.
In rclcpp's implementation, the group is constructed first and the
node references it afterward. At the tracepoint fire site, the
information is simply not available without reaching out to external
state.

IMPLICATION
-----------
Minor but worth noting:
  - CG → node association is derived, not directly traced.
  - If a CG is created but no entity is ever added to it AND the
    executor never claims it via add_callback_group_to_map, we cannot
    attribute it to a node. Such "orphan" CGs would not appear in the
    task model. In practice we have not observed any, but a future
    analysis that cares about this should add a tracepoint for
    Node::create_callback_group() specifically.


============================================================================
6. Callback groups shared across unexpected boundaries
============================================================================

EXPECTED
--------
A CG belongs to one node; the node's entities use it. Simple
containment.

OBSERVED
--------
Inside a single process, multiple entities across DIFFERENT nodes
sometimes share a CG when the process uses one CG as a "scheduling
bucket" rather than as a per-node grouping. In AAS this was not
observed because each process runs one primary node + service
bridges, but we have the data structure to detect it and expect it to
appear in composite containers (e.g., Autoware's sensing container
with 8+ composable nodes).

EVIDENCE
--------
Not yet in AAS — to be checked on the Autoware run.

IMPLICATION
-----------
The node → CG relationship is generally many-to-many in the formal
model. FISH records the CG as a first-class object with its own
identity; the relationship to nodes and entities is derived at
analysis time.

7. rclpy add_node / remove_node cycles cause naive tracepoint blow-up
============================================================================

EXPECTED
--------
"Per-process executor" intuition suggests that a scheduler-introspection
tracepoint placed at Executor.add_node should fire approximately once
per node per process lifetime — the same semantics as the rclcpp
`Executor::add_callback_group_to_map` fire site, which is guarded by
an atomic one-shot flag.

One might further assume that the rclpy Executor object itself is
destroyed and recreated on every spin cycle (since `rclpy.spin_once`
is a standalone function rather than a method on an executor instance).

Both intuitions are wrong, and they fail in opposite directions.

OBSERVED
--------
In the 2026-04-19 AAS mission (fish_compose_20260419_154439) with an
initial non-deduplicated `fish_rclcpp_executor_add_cbgroup` tracepoint patch
in rclpy, we measured:

  aircraft container total fish_rclcpp_executor_add_cbgroup events: 103,466
  yolo_py process (PID 1252) alone:                          ~25,000
  Distinct (executor_addr, group_addr, node_addr) triples in
    yolo_py:                                                         1

So the SAME logical association fired ~25,000 times inside yolo_py.

At the same time, the rclpy Executor was NOT being recreated:
  yolo_py (PID 1252) fish_rclcpp_executor_init events:                     1
  (SingleThreaded, 1 thread)

So exactly one executor lived for the entire yolo_py lifetime.

The ~25,000 add_cbgroup events came from the cycle inside
`rclpy.spin_once(node)`:

    def spin_once(node, executor=None, timeout_sec=None):
        executor = get_global_executor() if executor is None else executor
        node_was_added = False
        try:
            node_was_added = executor.add_node(node)
            executor.spin_once(timeout_sec=timeout_sec)
        finally:
            if node_was_added:
                executor.remove_node(node)

On each call, the executor's node set is EMPTIED (remove_node) and
then immediately REFILLED (add_node). From the `add_node`
implementation's perspective, each iteration looks like a brand-new
add:

    with self._nodes_lock:
        if node not in self._nodes:   # <-- True every iteration
            self._nodes.add(node)
            node.executor = self
            self._guard.trigger()
            trace_executor_add_cbgroup(...)   # fires every time
            return True
        return False

rclpy.spin_until_future_complete() and rclpy.spin() have the same
shape. yolo_py's inference loop burned ~25,000 spin_once cycles,
each firing our tracepoint once.

CONSEQUENCE
-----------
LTTng trace buffers are finite. 25,000+ events per yolo_py plus
similar volume from other rclpy PIDs produced ~100K redundant
fish_rclcpp_executor_add_cbgroup events across the aircraft trace, which
displaced other events: fish_rclcpp_publish_link events dropped from the
expected 88 to 1 in the first run. The trace was still usable for
graph extraction (32/33 entities got CG info via fish_rclcpp_cbgroup_add
which was not affected) but cross-container comm attribution via
fish_rclcpp_publish_link degraded from "88 genuine edges" to "1 residual".

ROOT CAUSE
----------
In rclcpp, `Executor::add_callback_group_to_map` refuses to add a
group twice:

    std::atomic_bool & has_executor = group_ptr->get_associated_with_executor_atomic();
    if (has_executor.exchange(true)) {
      throw std::runtime_error("Callback group has already been added to an executor.");
    }

So the C++ tracepoint cannot fire more than once per (executor,
group) pair — the structural guarantee comes from the source itself.

rclpy's Executor.add_node does not provide that guarantee (and
reasonably so: in rclpy the expected pattern is spin-helper driven).
The guarantee has to be added on the observation side.

FIX
---
Python-side dedup in the ctypes bridge: maintain a process-scoped
`_SEEN_EXEC_CG = set()` of `(id(executor), id(group), id(node))`
triples and short-circuit `trace_executor_add_cbgroup` if the triple
has already been reported. See notes/dedup_rclpy_exec.txt for the
full mechanism and invariants.

After the fix, fish_compose_20260419_161633 (identical mission):

  aircraft fish_rclcpp_executor_add_cbgroup: 103,466 → 34   (~3000× reduction)
  aircraft fish_rclcpp_publish_link:              1 → 89    (recovered to expected)
  yolo_py (PID 1252) fish_rclcpp_executor_init:              1 (unchanged)
  yolo_py EX vertex type:  ?  →  SingleThreaded, 1 thread, 1 CG
  9/9 EX vertices with executor_type + cb_groups (was 8/9)

IMPLICATION
-----------
This is a generic lesson for adding tracepoints to Python observability:
the C++ side and Python side may expose the "same" API but with very
different invariants. In rclcpp, `add_callback_group_to_map` is
one-shot by construction; in rclpy, `add_node` is callable in a
tight loop by design. Dedup belongs on the observation side.

See also:
  notes/rclpy_tracepoints.txt — prior observation about id() reuse
    across object destruction/recreation in CPython.
  notes/dedup_rclpy_exec.txt — full dedup mechanism + invariants.


============================================================================
8. Real GPU work often runs in a NON-executor thread
============================================================================

EXPECTED
--------
Classical "ROS 2 GPU node" intuition (and most task-model papers): the
GPU-hosting work — kernel launches, memory copies, inference — is
triggered by a callback. Typical mental model:

    subscription_callback(image_msg):
        preprocess(image_msg)
        kernels = launch_inference_kernels()   # GPU work
        publish_detections(kernels.output)

So the callback is the natural anchor for the GPU task: callback_start
marks the beginning, callback_end the end, and everything in between
(including all CUDA API calls and their kernels/memcpys) constitutes
the GPU DAG associated with that callback.

OBSERVED
--------
yolo_py in AAS (aircraft, PID 1252) runs its inference in the MAIN
Python thread, not a callback. The rclpy executor in the same process
handles only unrelated housekeeping.

Process structure (reconstructed from ros2_trace + cuda_runtime +
source inspection of yolo_node.py):

  yolo_py process, pid = 1252
  ├── main thread (tid = 1252)
  │     run_inference_loop() — infinite while loop:
  │        frame = frame_queue.get()
  │        boxes = self.do_yolo(frame)       # ONNX Runtime → 1M CUDA calls
  │        self.detection_publisher.publish(…)
  │     → 1,029,170 cuda_runtime events
  │     → 0   callback_start events
  ├── ros_spin_thread (tid = 1434)
  │     while rclpy.ok(): rclpy.spin_once(self, timeout_sec=0.001)
  │     → 0   cuda_runtime events
  │     → 59,407 callback_start events, all TimeSource.clock_callback
  └── frame_capture_thread (separate tid)
        cv2.VideoCapture.read() loop, pushes frames into the queue

Key measurements from the aircraft trace (fish_compose_20260419_161633):

  CUDA runtime events on tid 1252:        1,029,170
  callback_start events on tid 1252:              0   ← no callbacks!
  fish_rclpy_callback_register on pid 1252:            7
    (all TimeSource / ParameterService — nothing user-defined)
  fish_rclpy_subscription_callback_added on pid 1252:  1  (only /clock)

So the entire GPU task — 764K kernel launches, 78K memory copies, etc.
— is produced by a thread that is NOT managed by a ROS 2 executor.
There is no F vertex to attach a GPU DAG to, because no callback fires
for the work.

CAUSE
-----
rclpy.spin(node) / rclpy.spin_once(node) run synchronously: whatever
calls them owns the thread until they return. yolo_py wants an
asynchronous design — the main thread needs to pull video frames from
GStreamer at its own pace, not at a callback's pace. So the author
spun up `ros_spin_thread` as a dedicated thread for rclpy housekeeping
and kept the main thread free for the inference loop. Publishing
results via a rclpy Publisher does not require an executor.

This is a perfectly reasonable rclpy pattern — rclpy's Publisher is
thread-safe and can be called from any thread without executor
involvement. But it breaks the assumption that every scheduled entity
in a ROS 2 node is a callback.

FISH MODEL RESPONSE
-------------------
To keep FISH's hierarchy property intact (every layer N's expansion
yields exactly layer N+1 — CN→EX→N→E→F), we introduce two new
attribute values:

  etype = "oore"  (out-of-ROS entity)   — placeholder E vertex, one per
                                           oort thread, contains no aspects
                                           and no real ROS handles
  ptype = "oort"  (out-of-ROS thread)   — F vertex representing the
                                           thread itself; gpu_node=True
                                           when the thread issues CUDA
                                           API calls; DAG attaches here

For yolo_py the resulting sub-tree under the yolo_inference_node N:

  N "yolo_inference_node"
    ├── E etype="sub"   "/clock"                (real)
    │     └── F ptype="cpu" "TimeSource.clock_callback"
    ├── E etype="serv"  "/list_parameters"      (real)
    │     └── F ...
    ├── … (other parameter services)
    └── E etype="oore"  "tid_1252_io"           (NEW placeholder)
          └── F ptype="oort" "tid_1252_gpu_loop"  (NEW, gpu_node=True)
                 gpu_dags = […]                   (attached by gpu_dag.py)

Detection (postprocess/model_improved.detect_oort_threads):

  For each GPU-classified process p (via nsys_profiled_pids):
    cuda_tids  = set of `tid` in cuda_runtime for (session, container)
    cb_tids    = set of `vtid` firing callback_start under pid p
    oort_tids  = cuda_tids − cb_tids          # CUDA activity, no callbacks
    For each t in oort_tids:
       create oore (E) + oort (F) under p's N vertex.

IMPLICATION FOR SCHEDULING ANALYSIS
-----------------------------------
  - oort threads are tasks just like callback-triggered callbacks,
    but their scheduling unit is "an iteration of the main loop",
    not "an executor dispatch of a callback".
  - Period is determined by the rate at which frames arrive (sensor
    rate) — the loop naturally blocks on frame_queue.get().
  - They compete for CPU with the executor's threads (same process,
    same ready-list under Linux), so classical WCRT still needs them.
  - Separate from multi-executor-per-process (§1) — there is only ONE
    rclpy executor in yolo_py's process. The oort thread is outside
    the executor entirely.

This pattern likely appears in any ROS 2 node whose main work is
driven by a non-ROS input source: camera frames, hardware sensors
read via their vendor SDK, network streams decoded by GStreamer, etc.
Autoware's perception nodes (message-driven) should NOT exhibit this;
yolo-style apps (camera-driven) WILL.


============================================================================
9. GPU-side profiling gives us CPU-caller info, but only at sync points
============================================================================

EXPECTED
--------
To attribute a CUDA kernel launch back to user code (e.g. "this
`launchKernel` came from `onnxruntime.InferenceSession.run`"), the
natural assumption is that nsys captures a call stack for every
CUDA API call the same way it captures kernel activity.

OBSERVED
--------
nsys records backtraces only for a small subset of CUDA API calls.
In the aircraft trace:

  CUPTI_ACTIVITY_KIND_RUNTIME total rows:       1,029,170
  rows with non-zero callchainId:                   4,138   (~0.4 %)

Every one of those 4138 backtraces is for a `cudaStreamSynchronize*`
call. The 1M+ `cudaLaunchKernel` / `cudaMemcpyAsync` calls have
callchainId=0 — no backtrace recorded.

CAUSE
-----
nsys's CUDA backtrace capture is opt-in and sampled. Without
`--cudabacktrace=true` (or equivalent) at profile time, only certain
"sync-class" CUDA API calls get a stack. FISH's GPU daemon currently
starts nsys with default flags, which do not enable dense CUDA
backtrace.

Even when captured, resolution is partial:
  - Frames inside libcupti / libcudart: resolved by NVIDIA's bundled
    symbols (e.g. `cudaStreamSynchronize`).
  - Frames inside libonnxruntime_providers_cuda.so: unresolved,
    shown as raw hex addresses.
  - Python interpreter frames: not captured without `--python-backtrace`.

So the ingested data today cannot tell us "this cudaLaunchKernel
was called from do_yolo → ONNX Runtime → conv layer N". The best we
have at the moment is the libcudart api_name column plus a partial
call stack for sync points.

FISH RESPONSE
-------------
Two parallel tracks (tracked in notes/immediate_work.txt #0):

  (a) Extend nsys invocation in the FISH GPU daemon to enable
      `--cudabacktrace=true --python-backtrace=cuda` (approximate,
      exact flags depend on nsys version).
  (b) Extend postprocess/ingest.py to pull CUDA_CALLCHAINS + StringIds
      from the nsys sqlite and attach the resolved frame symbols to
      each cuda_runtime row in InfluxDB (as a tag or separate
      measurement). This is a one-way extension; existing callchainId=0
      rows stay unenriched until re-traced.

Without (a), (b) still helps for the 0.4 % of calls that do have
backtraces — sync points are meaningful boundaries in the DAG.


============================================================================
10. "One GPU task = one DAG" is not observed even in a simple workload
============================================================================

EXPECTED
--------
A deterministic, pre-compiled GPU inference pipeline (yolo26n ONNX
fed fixed-size 320×320 frames) should produce IDENTICAL CUDA API
call sequences every iteration — same kernels, same launch order,
same memcpys. Therefore the Typed DAG per invocation should also
be identical, and the "set of unique DAGs per oort thread" should
have size 1.

The paper's task model §4.7 algorithm hinges on extracting the
"longest period" DAG; the assumption was that variability across
invocations is rare.

OBSERVED
--------
Per-invocation DAG extraction on yolo_py over 1,980 inference
invocations (fish_compose_20260419_161633, aircraft, tid=1252,
gap>10ms period threshold) yielded:

  1,980 invocations
     44 unique DAG shapes (by canonical signature over (pe, name)
                           sequence + edge set, ignoring durations)

Distribution:

  shape  count  fraction   description
  ─────  ─────  ────────   ─────────────────────────────────────
  [0]    1,818   91.8 %   canonical yolo26n pipeline:
                          1050 nodes (550 CPU + 458 SM + 42 CE),
                          1548 edges
  [1]       99    5.0 %   2 CPU nodes only (cudaStreamSynchronize
                          ×2) — idle main-loop tick (no frame
                          available, thread blocks on queue.get)
  [2]       12    0.6 %   "double pipeline" — 2100 nodes, 2× shape
                          [0]. Artifact of the gap>10ms period
                          detector when two consecutive frames hit
                          back-to-back with <10ms idle between.
  [3]        5    0.25%   shape [0] minus 5 CPU nodes — likely a
                          small init-path branch skipped
  [4..43]   ~46    2.4 %   assorted minor variations (extra launch
                          here, missing memset there). Many at
                          count=1-2.

CAUSE (initial diagnosis — later refined, see UPDATE below)
-------------------------------------------------------------
Two distinct sources of variability:

  (a) Detector artifact.  Shape [2] (12 invocations) is almost
      certainly two frames stuck together because the real inter-
      frame idle dropped below the 10ms gap threshold.

  (b) Runtime non-determinism in ONNX Runtime — minor per-frame
      variations in kernel launch count.

UPDATE (2026-04-19 late) — pattern-based period detection
----------------------------------------------------------
The original period detection was time-gap-based: any gap
> gap_ms_threshold ends a period. This over-counted unique shapes
because it arbitrarily cut sequences that happened to have a brief
quiet moment (e.g., a short memcpy wait during the middle of an
inference). Most "unique" shapes in the 44 were actually noise
from this arbitrary slicing.

The replacement detector (postprocess/gpu_dag.py) uses a PATTERN:

    a period ends at a cudaStreamSynchronize* whose gap to the
    next CUDA API call exceeds gap_ms (default 5ms)

This matches the actual shape of a yolo inference cycle:
    [boundary sync] (frame-wait idle) → cudaMemcpyAsync (H2D) →
    many cudaLaunchKernels + cudaMemcpyAsyncs → final
    cudaStreamSynchronize (next boundary).

After the switch, the same 1980-ish invocations yield:

    6 unique DAG shapes
    [0] 1,857 invocations (99.7%)  1050 nodes (550 CPU + 458 SM + 42 CE), 1548 edges
        ← canonical yolo26n inference, matches expected determinism
    [1]     2 invocations (0.1%)   "double" pipeline (edge case: two
                                   inferences still with no idle between)
    [2]-[5] 1 invocation each       startup / shutdown / tail artifacts

For scheduling-analysis purposes, shape [0] is effectively THE task
DAG. The four 1-count outliers at init/teardown are not part of the
steady-state task set.

Idle-filter: periods with fewer than 2 GPU ops (min_gpu_ops=2) are
dropped — these are no-work spans where the main loop was simply
polling the frame queue.

LESSON FOR FISH
---------------
  - Time-gap period detection is too blunt for heterogeneous
    workloads. Use workload-specific pattern anchors (stream sync,
    NVTX ranges, or explicit application markers) to get
    deterministic period boundaries.
  - When the pattern holds, observed DAG variance drops by orders
    of magnitude (44 → 6 here, dominant shape coverage 91.8% →
    99.7%). The residual variance is genuine ONNX Runtime non-
    determinism plus startup/shutdown tails — those are expected
    in real systems, not detector bugs.

IMPLICATION FOR SCHEDULING ANALYSIS
-----------------------------------
  - The Typed DAG task model (PDF §4.5) needs a notion of
    "dominant DAG" (covers ≥95% of invocations) plus optional
    "secondary DAGs" for robust WCRT analysis. In yolo_py, the
    dominant shape [0] covers 91.8% — treating it as the single
    task DAG is defensible but gives a small tail-latency blind
    spot for shapes [2]-[4].
  - For roofline/WCET analysis, aggregate the per-node c_min_ns /
    c_max_ns within the dominant shape (we do this already —
    gpu_dag.py tracks BCET/WCET per node across invocations that
    match the same signature).
  - The "idle" shape [1] is NOT a task — it's a gap where the main
    loop was waiting for input. Future gpu_dag.py revisions should
    filter these (threshold on total DAG compute time, or require
    ≥1 SM/CE node to qualify).

FISH RESPONSE
-------------
postprocess/gpu_dag.py already produces the full unique-DAG set
with per-node c_min/c_max, signatures, and invocation counts. No
code collapses the set into "one dominant DAG" — downstream
analysis can pick the policy (dominant-only, top-k, all).

Pipeline:
  1. detect_oort_threads creates F ptype="oort" under N (step done).
  2. gpu_dag.py per-F: period-detect CUDA bursts on the thread,

     (step done).
  3. fish_compose --from-cache propagates per-container graphs into

Next improvements (not yet done):
  - Filter shape [1]-like "idle" DAGs (no SM/CE nodes).
  - Tune period threshold or switch to sync-based boundaries to

  - When CUDA backtraces arrive (§9 → immediate_work 0a), enrich

============================================================================
Summary: why this matters for the paper
============================================================================

Every scheduling-analysis paper we have seen on ROS 2 uses one of:
  - A textbook Executor model (1 process, 1 executor, 1 CG per node,
    SingleThreaded OR MultiThreaded but not mixed)
  - A pure callback-graph model that ignores executors and CGs entirely

Neither captures:
  - Multiple executors per process (both transient and persistent)
  - Mixed MutuallyExclusive + Reentrant CGs within one node
  - Runtime-determined thread count via hardware_concurrency()
  - The rclpy vs rclcpp divide

FISH's contribution is a task model derived from observation rather
than design documents, where each of these deviations is captured and
measurable. The AAS results already demonstrate that real ROS 2
applications violate every common simplification.

Paper framing for the "problem" section:
  1. Quote the standard executor model from the ROS 2 Foxy / Humble
     design notes (or Casini et al. ECRTS 2019 model).
  2. Show the AAS measurements above as counter-examples.
  3. Argue that response-time analysis needs the observed model.

Paper framing for the "approach" section:
  1. FISH captures executor type + thread count + CG type + CG
     membership with zero source-code modification to the application.
  2. The graph_store persists this as mutable graph state so
     downstream WCET, WCRT, GPU contention analyses can accumulate.

Appendix B. Open questions for future measurement
============================================================================

1. Transient vs persistent classification.
   Can we distinguish them by executor lifetime? Adding a tracepoint at
   Executor destruction would give us per-executor lifetime. Very-short
   lifetimes (< 1s) almost certainly correspond to spin() / spin_some()
   helpers.

2. Spin() without add_node.
   rclcpp::spin(node) creates an executor AND calls add_node internally.
   But a user's own Executor might be constructed without any add_node
   call yet. Do we observe such executors? (Would show up as
   fish_rclcpp_executor_init with zero matching fish_rclcpp_executor_add_cbgroup.)

3. CG reuse across nodes.
   Does any AAS process share a single CG across multiple nodes? Check
   with: for each group_addr, count distinct node_addrs it's associated
   with via fish_rclcpp_executor_add_cbgroup or inferred from entities.

4. Reentrant CG concurrency actually observed.
   Just because a CG is Reentrant does not mean the callback ever runs
   concurrent instances. Cross-check callback_start/end timings: for
   each Reentrant CG, find cases where two callback_start of the same
   callback overlap in time (across vtids).

5. rclpy behavior.
   After installing rclpy scheduler tracepoints, re-measure yolo_py and
   compare: MT in rclpy != MT in rclcpp due to GIL.
