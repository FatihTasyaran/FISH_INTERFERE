Shape classification via embedding — starting notes
=====================================================

Motivation:
  The current phase detector (notes/phase_detector.txt) categorises
  each period as INIT / WARMUP / STEADY / TAIL / COALESCED / UNKNOWN.
  On yolo the UNKNOWN bucket is empty; on Autoware it is 73 %, but
  most of those UNKNOWNs differ from the dominant shape by only 1-3
  nodes. That variance is the interesting signal — it tells us which
  driver-level / kernel-level if-else branches are being taken — and
  a plain "UNKNOWN" label throws it away.

  The goal is a classification that preserves the variance structure
  AND produces a small, interpretable taxonomy.


Proposed classification (replacing UNKNOWN with a richer scheme)
----------------------------------------------------------------

  shapes = [INIT, FINISH, DOMINANT, variations...]

    INIT       — lifecycle, one-shot CUDA-context / engine-load work
                 fires at process start (init_score ≥ 100).

    FINISH     — lifecycle, late-run truncation where the process
                 dies mid-invocation; sequence is a near-prefix of
                 DOMINANT with trailing sync-only cleanup.

    DOMINANT   — the mode of signatures across productive periods;
                 the paper's "canonical task DAG" per pipeline stage.

    variations — every other productive period. NOT a single bucket
                 but a STRUCTURED SET: each variation has a
                 similarity-to-DOMINANT score and possibly sub-cluster
                 membership.

  INIT and FINISH detection stay score-based / position-based (fast,
  explicit). DOMINANT is the signature mode. The novelty is in how we
  characterise `variations`.


Why embedding
--------------

Variations arise from:

  (a) Driver-level if-else: cuBLAS / cuDNN pick different kernels
      based on heuristics (input sizes, alignment, workspace budget).
      Result: 1-3 node substitutions, same macro structure.

  (b) TRT execution context state: first N calls after an engine
      binding change may fall through different paths (algorithm
      auto-select, tactics replay). Result: occasional extra kernels
      that vanish after warmup.

  (c) Application-level if-else: a callback conditionally skips a
      memcpy, or dispatches a different post-processing head.
      Result: whole subsequences appear or disappear.

  (d) Detector artifacts: period boundary missed or spurious; shape
      inflates (coalesced) or truncates.

Each category has a distinct similarity profile:
  (a) → high similarity to DOMINANT, edit-distance ≤ 3
  (b) → transient cluster that shrinks with more data
  (c) → one or two distinct mini-clusters at moderate distance
  (d) → outliers OR multiples of DOMINANT (coalesced)

A distance-based embedding over (pe, name) sequences separates these
cleanly and gives an interpretable figure for the paper.


Proposed similarity metric
--------------------------

Primary: Levenshtein edit distance on the node sequence
         node_seq(P) = [(pe_i, name_i) for each node i in P].

  - Substitution cost 1 for (pe, name) tuple swap (covers case (a)).
  - Insertion / deletion cost 1 each (covers case (c)).
  - Distance 0 ⟺ identical signature (DOMINANT by definition).

Rationale:
  Our DAGs are acyclic, and every edge is forward in issuance order,
  so the node sequence is a topological linearisation. Two shapes
  with the same multiset of nodes in the same issuance order have
  the same precedence DAG (stream FIFO + cpu_order edges are
  determined by the sequence). Levenshtein on the sequence captures
  the structural difference faithfully.

Alternatives considered:
  - Graph edit distance: overkill; our DAGs are essentially paths.
  - Feature vector (counts per kernel name + totals): loses order
    information; would conflate "H2D, kernel, D2H" with "D2H, kernel,
    H2D" which ARE different tasks.
  - Weisfeiler-Lehman kernel: good for unordered graphs; again, not
    needed when order matters and the DAG is near-linear.
  - node2vec / graph2vec: require training, parameter tuning, and
    give un-interpretable vectors. Bad fit for a paper figure.

Normalisation:
  Report BOTH raw edit distance AND normalised distance
    d_norm(P, Q) = edit(P, Q) / max(|P|, |Q|)
  Normalised distance ∈ [0, 1] is comparable across workloads with
  different pipeline sizes.


Pipeline (what a run of embed-analysis does)
--------------------------------------------

  1. Input: set of productive periods from gpu_dag.py (per
     stream/process, sync-to-sync or gap-based spans).

  2. For each period P:
       s(P) = list of (pe, name) tuples in issuance order
       label(P) ∈ {INIT, FINISH, None} by score/position rules.

  3. For unlabelled periods:
       sig(P) = sha1(repr(s(P)))
       mode sig is DOMINANT.
       Periods with sig == DOMINANT get label DOMINANT.

  4. For the remaining periods (variations):
       compute pairwise edit distance matrix D[i][j] = edit(s(P_i), s(P_j))
       compute distance-to-dominant: d0[i] = edit(s(P_i), s(DOMINANT))

  5. Cluster:
       agglomerative clustering, linkage='average',
       distance_threshold = 3  (≤ 3 node differences = same cluster).
       → each variation ∈ one mini-cluster; clusters get letters
         A, B, C, ... in descending size.

  6. Report:
       a. A summary line:
          "5 phases: INIT=2, FINISH=1, DOMINANT=1857 (coverage 96.1%),
           variations in 3 clusters [A:72, B:24, C:8]".
       b. A distance-to-DOMINANT histogram (binned at 0, 1, 2, 3, 4+)
          showing the "how close are variations" question directly.
       c. Per-cluster exemplar: one sample DAG from each variation
          cluster, with its diff against DOMINANT highlighted (the
          1-3 nodes that differ).

  7. Figures for paper:
       F1. Shape-distance histogram (x = normalised edit distance,
           y = count). One bar at 0 (DOMINANT), long tail for
           variations.
       F2. 2D MDS projection of the distance matrix, coloured by
           cluster. Shows variation cluster structure at a glance.
       F3. Exemplar diff: DOMINANT's node sequence vs the top-2
           variation cluster exemplars. Visualises which kernels
           flipped between branches.


Interpretation rules (for automated reporting)
----------------------------------------------

  - If all variations have d_norm ≤ 0.05 AND cluster count ≤ 2:
      "workload has a stable canonical task with driver-level
       branch perturbations" (case a).
  - If clusters are distinct (d_norm > 0.15 between clusters) and
    at least one has count ≥ 10:
      "workload has multiple task variants — application-level
       branching" (case c). Report each variant.
  - If a single cluster dominates but drops off sharply in recency:
      "workload is still warming up" (case b). Flag for longer
       measurement run.
  - If variations include |P| > 1.5 × |DOMINANT| and sequence
    matches N copies of DOMINANT:
      COALESCED(N) (case d, existing rule).
  - Residual outliers: log verbosely; they drive the next iteration
    of rule refinements.


Open questions
--------------

  1. Edit-distance weights: should substituting one SM kernel for
     another cost 1 uniformly, or should similar kernels (both
     GEMM, both pooling) get a lower cost? Tentative: uniform 1
     for v1; refine if cluster-separation is noisy.

  2. FINISH detection granularity: current rule requires match_len
     / |P| ≥ 0.95 and trailing-all-sync. With edit distance, a
     relaxed rule "d_norm ≤ 0.10 AND P in last 3 periods" is
     simpler and may suffice.

  3. DOMINANT ambiguity: what if no single signature has ≥ 50 %
     coverage? Pick the largest cluster centre (after clustering)
     instead of the raw mode.

  4. Computational cost: pairwise edit distance is O(N² · L²) where
     N = # periods, L = avg sequence length. On yolo (N≈1860,
     L≈1000), that's ~10⁹ ops — slow in pure Python. Mitigations:
     (a) bucket periods by |length| difference; only compare pairs
         whose lengths are within a tolerance;
     (b) Rust extension via rapidfuzz or polyleven (both expose
         bounded edit distance).

  5. Cross-workload embedding: can we project yolo's DOMINANT into
     Autoware's DOMINANT space? Not v1; first establish within-
     workload fidelity.
