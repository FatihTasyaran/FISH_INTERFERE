ROS 2 Action Decomposition for FISH Model
==========================================

1. Background
-------------
ROS 2 Actions are a higher-level communication pattern built on top of
services and topics. Unlike simple request-response (services) or
fire-and-forget (pub/sub), actions support long-running tasks with
feedback and cancellation. This makes them critical for real-time
autonomous systems (e.g., takeoff, land, orbit maneuvers).

In the real-time systems literature, actions are typically ignored or
hand-waved because their composite nature makes them difficult to model.
FISH explicitly decomposes actions into their constituent primitives,
preserving the scheduling-observable behavior while maintaining
traceability to the higher-level action semantics.


2. Action Decomposition
-----------------------
Every ROS 2 action server is implemented as exactly 5 communication
primitives (this is enforced by the rclcpp framework, not convention):

  Action Server = 3 Services + 2 Publishers

  Services (request-response):
    /<action_name>/_action/send_goal      — client sends a goal
    /<action_name>/_action/cancel_goal    — client requests cancellation
    /<action_name>/_action/get_result     — client retrieves final result

  Publishers (periodic/event-driven):
    /<action_name>/_action/feedback       — server publishes progress
    /<action_name>/_action/status         — server publishes goal status
                                            (GoalStatus: ACCEPTED,
                                             EXECUTING, CANCELING,
                                             SUCCEEDED, CANCELED, ABORTED)

The naming pattern /<name>/_action/<component> is hardcoded by the
ROS 2 action infrastructure and is universally reliable for detection.

4. FISH Model Representation
-----------------------------
In the FISH graph, an action is an Entity vertex (level 2) with:

  Entity type:  "act" (action)
  A_v:          {label: action_name, etype: "act",
                 action_components: {
                   "send_goal":   <service_entity_id>,
                   "cancel_goal": <service_entity_id>,
                   "get_result":  <service_entity_id>,
                   "feedback":    <publisher_entity_id>,
                   "status":      <publisher_entity_id>
                 }}
  Z_v:          [function_ids...]  (callback functions triggered by
                                    any of the 5 components)

The action is an "abstract entity" — a grouping node whose Z_v
contains the entity IDs of its 5 components. The 5 component entities
(3 services + 2 publishers) ALSO exist as individual Entity vertices
in the graph with their own Function children. They are NOT removed
from the flat service/publisher lists; the action entity references
them without replacing them.

This design serves multiple purposes:

  (a) Preserves the semantic meaning of the action (e.g., "land_action"
      is a coherent operation, not 5 unrelated endpoints)
  (b) Enables action-level timing analysis (worst-case from send_goal
      to get_result, feedback periodicity, cancellation latency)
  (c) Maintains decomposition transparency — expanding the action
      Entity reveals its constituent services and topics
  (d) Supports the FISH interactive visualization: clicking an action
      entity expands it to show its 5 components, exactly like
      clicking an executor expands to show its nodes

This is consistent with the FISH expansion operator: at the collapsed
level, the action appears as a single entity; upon expansion, the 5
primitives and their individual scheduling functions become visible.
Component entities remain independently addressable in the graph for
per-primitive analysis.


5. Detection Algorithm
----------------------
Given a set of services and publishers for a node:

  1. Collect all names containing "/_action/"
  2. Extract action base names: name.split("/_action/")[0]
  3. For each unique base name, verify all 5 components exist:
     - send_goal, cancel_goal, get_result (services)
     - feedback, status (publishers)
  4. If all 5 present: create abstract Action entity, keep components
     in flat service/publisher lists, reference them via action_components
  5. If partial (< 5): flag as WARNING — incomplete action, keep
     components as individual entities only

This algorithm works identically on node_info data (where actions may
already be in Action_Servers) and on trace-derived fallback data (where
actions appear as flat services and publishers).


6. Implementation (verified, session fish_20260329_182842)
----------------------------------------------------------
identify_entities() creates the following structure per action:

  Action entity (etype="act", level 2):
    A_v:
      label:             "/Drone1/land_action"
      action_type:       "autopilot_interface_msgs/action/Land"
      action_handle:     0x61EA75ED11B0  (from fish_rclcpp_action_server_init)
      action_components: {
        send_goal:    <entity_id>,       # service component entity
        cancel_goal:  <entity_id>,       # service component entity
        get_result:   <entity_id>,       # service component entity
        feedback:     <publisher_handle>,
        status:       <publisher_handle>
      }
    Z_v: []  (empty — abstract entity, functions live on components)

  3 service component entities (etype="serv", level 2):
    Each has: label, service_handle (from fish_rclcpp_action_server_init),
              action_role ("send_goal"/"cancel_goal"/"get_result")
    Added to parent node's Z_v as independent entities.

  2 publisher components stored as handles in action_components
    (not separate entities — publishers have no callbacks).

identify_callbacks() creates function vertices on the component
service entities (NOT on the action entity):

  Component entity: /Drone1/land_action/_action/send_goal
    Z_v: [function_id]
    Function: land_action::goal (ptype="cpu")

  Component entity: /Drone1/land_action/_action/cancel_goal
    Z_v: [function_id]
    Function: land_action::cancel (ptype="cpu")

  Component entity: /Drone1/land_action/_action/get_result
    Z_v: [function_id]
    Function: land_action::result (ptype="cpu")

Key insight: action component services do NOT fire
rclcpp_service_callback_added. They use internal dispatch via
ServerBase::execute_goal_request_received() etc. The function
vertices are created structurally from the action_role attribute.
Runtime timing will come from fish_rclcpp_action_execute_goal/cancel/result
custom tracepoints (FISH) which carry action_server_handle.

Verified counts for px4_interface node (4 actions):
  Entities: +12 component services, +4 action abstracts = +16
  Functions: +12 (4 actions × 3 handlers)
  Total model: 5 executors, 6 nodes, 49 entities, 45 functions
