FISH use-case suite — planning
==============================

Date: 2026-05-17

Purpose
-------
A small, deliberate set of workloads FISH is validated against. The
suite serves three goals at once:

  1. README "Tested on" credibility — show breadth (rclcpp / rclpy /
     multi-container / GPU-heavy / GPU-light).
  2. Paper Results section evidence — load-bearing numbers cite
     these sessions by name.
  3. CI/CD test surface — small variants of each become fixtures
     (tests/fixtures/<name>/) for the smoke + contract test tiers.

Each use-case is captured ONCE end-to-end, the trimmed trace lives
in the repo (Git LFS for the > 50 MB ones), and paper_anchors.json
locks in the load-bearing counts derived from it. Drift on those
counts is what the CI/CD smoke job tests.

Planned additions (release-readiness)
-------------------------------------

D. Isaac ROS AprilTag (NEW, planned 2026-05-17)
   Why this one:
     - Well-known NVIDIA reference workload — anyone in the ROS 2 +
       GPU community has heard of Isaac ROS, so claiming
       "FISH profiles Isaac ROS AprilTag end-to-end" is a credible
       single-line elevator pitch.
     - Compact: one container, one camera input, one detection node.
       Trace under 30 s captures the whole steady-state pipeline.
     - Pure GPU pipeline — every callback either does CUDA work or
       feeds something that does. Stream-bridge attribution (Rule A
       in particular) and per-token GPU statistics are showcased
       cleanly.
     - Deterministic input (rosbag replay) — repeatable across runs,

CI/CD wiring
------------

For each use-case in the "Current" + "Planned" sections:
  1. Capture session → trim trace → commit fixture
  2. Add a paper_anchors.json entry with the load-bearing counts
  3. Add a smoke test that runs the full pipeline against the
     fixture and asserts those counts (with a small tolerance)

The smoke job in .github/workflows/ci.yml runs on a schedule +
release-prep branch trigger (see immediate_work.txt #15). Each
use-case adds maybe 30 s to that job. Five use-cases × 30 s =
~2.5 min total — comfortable within free-tier minutes.

What NOT to put in the suite:
  - Synthetic micro-benchmarks (one kernel, one callback) — no
    paper credibility, and they don't surface integration bugs.
  - Anything that requires recompiling a 3rd-party stack from
    source as part of CI — set-up time dominates.
  - Anything that needs network access during CI (downloading
    large weights, fetching map tiles). Bake the inputs into the
    fixture.

Strategic dependency: NVIDIA-ISAAC-ROS/ros2_benchmark
===============================================================================
Date added: 2026-05-17

The repository https://github.com/NVIDIA-ISAAC-ROS/ros2_benchmark is
a substantial gift for the FISH use-case suite. Three independently
valuable axes:

Axis 1 — rosbag + use-case source
---------------------------------
Hosted on NVIDIA NGC as the `r2b` dataset family.

  r2b 2024 (~2.9 GB total, 4 sequences):
    r2b_galileo        (471 MB)  room w/ 4× LI HAWK stereo + objects
    r2b_galileo2       (986 MB)  same env, RealSense D455
    r2b_robotarm       (1.4 GB)  robot arm + tray-with-objects
    r2b_whitetunnel    (503 MB)  hallway, HAWK cameras

  r2b 2023 (~15.8 GB total, 8 sequences):
    r2b_lounge / storage / hallway / datacenter / cafe / hideaway /
    mezzanine (1.2–3.9 GB each), r2b_hope (30 MB, single 6-DoF pose
    estimation image)

Download:
    ./ngc-cli/ngc registry resource download-version \
        "nvidia/isaac/r2bdataset2023:3"

These are paper-cited datasets. "Validated on the r2b dataset" is a
single-line credibility win.

Axis 2 — ground-truth graph topology
------------------------------------
Every ros2_benchmark workload runs under a known 4-node template:

    ComposableNodeContainer (component_container_mt)
      ├── DataLoaderNode          ← reads rosbag
      ├── PlaybackNode            ← controls publish rate
      ├── <workload-specific>     ← AprilTag / Stereo / etc.
      └── MonitorNode             ← collects latency / fps / jitter

Single-node scripts in their /scripts/ directory:
    apriltag_ros_apriltag_node.py
    image_proc_rectify_node.py
    image_transport_h264_encoder_node.py
    image_transport_h264_decoder_node.py
    stereo_image_proc_node.py
    realsense2_node.py

Multi-node graph scripts:
    apriltag_ros_apriltag_graph.py
    stereo_image_proc_graph.py

For each script the EXACT (N, E, F) cardinality is derivable from the
launch file. This becomes the strongest possible ground truth FISH
can be tested against: "FISH extracted X nodes — the benchmark's
launch defined exactly X nodes."

Status: tests/integration/test_graph_ground_truth.py is task #52
(pending).

Axis 3 — overhead measurement (paper-required)
----------------------------------------------
MonitorNode emits a stable JSON metric block per run:

    MEAN_FRAME_RATE             (fps)
    FIRST_SENT_RECEIVED_LATENCY (ms)
    LAST_SENT_RECEIVED_LATENCY  (ms)
    MAX_JITTER, MEAN_JITTER     (ms)
    NUM_MISSED_FRAMES           (count)
    RECEIVED_DURATION           (ms)
    MAX_CPU_UTIL, MEAN_CPU_UTIL (%)
    BASELINE_CPU_UTIL           (%)

A/B protocol — for each benchmark workload:

  1. Run vanilla:          FISH_ENABLED=0  ros2 launch <bench>
  2. Run FISH-enabled:     FISH_ENABLED=1  ros2 launch <bench>
  3. Diff the JSON outputs → fps / latency / jitter / dropped-frames
     deltas attributable to FISH wrapping (LTTng + nsys + GPU daemon).

Reference values to beat:
  AprilTag node: 90.8 fps @ 11 ms latency on Intel NUC
  AprilTag node: 56.3 fps @ 18 ms latency on AGX Orin

If FISH adds ≤ 5 % to mean frame rate degradation and < 1 ms to mean
latency on the AGX Orin numbers, the paper's overhead claim is
defensible. If not, we know where to look.

Status: paper Verification section data; task #51 (pending).

Complementary narrative for the paper
-------------------------------------
ros2_benchmark answers:  "Is this pipeline fast enough?"
FISH answers:            "Where is the time going?"

So when ros2_benchmark reports a benchmark missing its SLA, FISH can
break the result down per-callback and per-skeleton-token. The two
tools live in the same paper without competing.

Practical wiring for the use-case suite
---------------------------------------
The Isaac ROS AprilTag entry above (use-case D) gets retargeted to
use ros2_benchmark's apriltag_ros_apriltag_node.py rather than a
hand-rolled launch. Mission YAML examples/isaac_apriltag.yaml will
be updated once we verify the ros2_benchmark setup runs inside the
FISH-baked Isaac ROS image. (Currently the YAML targets the
isaac_ros_apriltag stand-alone pipeline; switching to ros2_benchmark
gives us the MonitorNode metrics for free.)
