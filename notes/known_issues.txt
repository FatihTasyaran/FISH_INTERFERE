ISSUE-001: rclpy use_sim_time timer recreation causes massive event inflation
-----------------------------------------------------------------------------
Date discovered: 2026-03-29
Session: fish_20260329_182842

Observation:
  rclpy_timer_callback_added fires 88,789 times in a ~30 second session
  for yolo_inference_node (PID 952). There are hundreds of distinct
  timer_handle values, many repeating ~3000 times each (memory reuse).

  This matches the rcl_timer_init pattern from the previous session
  (fish_20260322_212915): 166,890 rcl_timer_init events for 5 actual
  timers. The periodic re-init is a known rclpy/rcl behavior when
  use_sim_time is enabled.

Root cause (verified in rclpy source):
  rclpy/executors.py _wait_for_ready_callbacks() (line 510) creates a
  NEW Timer object on EVERY call when timeout_sec > 0:

    timeout_timer = Timer(None, None, timeout_nsec, self._clock, ...)

  This is called from spin_once() → _spin_once_impl() → wait_for_ready_callbacks()
  on every executor spin cycle. A node with a 1ms timer spins ~3000/s,
  creating ~3000 Timer objects per second. In 30 seconds: ~88K Timers.

  Each Timer.__init__() calls _rclpy.Timer() which calls rcl_timer_init()
  in the C layer, firing the rcl_timer_init LTTng tracepoint. Our
  rclpy_timer_callback_added tracepoint fires from the same __init__.

  The 348 distinct timer_handle values with high repeat counts (top handle:
  3376 times) confirm memory reuse after free — the rcl_timer_t is
  allocated, used for one WaitSet cycle, then destroyed, and a new one
  is allocated at the same address.

  Note: rclpy_timer_link_node count is 0 for PID 952, confirming these
  are NOT user-created timers (Node.create_timer). They are executor-
  internal timeout timers with callback=None.

  time_source.py was also checked — it does NOT recreate timers. It only
  calls clock.set_ros_time_override() on each /clock message.

Impact:
  - Data volume: 88K unnecessary timer events in a 30s session. Scales
    linearly with session duration and sim clock rate.
  - Cache/TLB pollution: each Timer.__init__() allocates a new rcl_timer_t,
    causing memory churn. The ctypes bridge call adds a small overhead per
    recreation but the dominant cost is rclpy's own allocation pattern.
  - Ingest time: contributes to the 8.6M event total and 12min ingest.
  - Model extraction: not affected — find_one() returns first match, which
    is the initial creation. Subsequent recreations are ignored.

Upstream concern:
  The executor creates and destroys a Timer object on every spin cycle
  just to implement a wait timeout. This is wasteful:
    - Unnecessary malloc/free cycles (~3000/s per node)
    - Cache line invalidation from repeated allocation
    - LTTng tracepoint overhead (rcl_timer_init fires on each creation)
    - Potential scheduling jitter from allocation latency
  A reusable timeout timer (create once, reset period as needed) would
  eliminate this overhead entirely. rclcpp does not have this issue.

FISH mitigation options:
  1. Filter in tracepoint: check if timer_handle was already seen for this
     node before firing rclpy_timer_callback_added. Adds complexity to the
     ctypes bridge but eliminates 99.9% of redundant events.
  2. Filter at ingest: deduplicate timer events by (vpid, timer_handle)
     keeping only the first occurrence. Simple but doesn't reduce trace
     data volume on disk.
  3. Accept and document: model extraction already handles it correctly.
     The extra events are noise but don't cause incorrect results. Address
     as part of data volume optimization (immediate_work.txt item 5).

Current status: option 3 (accepted, documented).


ISSUE-002: nsys `utcEpochNs` is actually LOCAL-epoch-ns (nsys/CUDA quirk)
------------------------------------------------------------------------
Date discovered: 2026-04-20
Session: fish_20260419_211621

Observation:
  Any analysis that correlates nsys GPU timestamps with MongoDB
  ros2_trace callback timestamps returns wildly misaligned results.
  Example on a Europe/Amsterdam (UTC+2) machine:

    nsys session_start (col labelled utcEpochNs): 1776633420939599477 ns
      → naïvely interpreted as UTC, decodes to 2026-04-19 21:17:00 UTC
    First ros2:callback_start (correct UTC):      1776626222756852919 ns
      → decodes to 2026-04-19 19:17:02 UTC

  Delta: 7200 s = exactly the Amsterdam UTC offset. Naïve time-window
  membership then reports 100 % of CUDA calls "outside any callback
  window" — because the CUDA timestamps are shifted 2 h forward
  relative to the LTTng ones.

Root cause:
  `TARGET_INFO_SESSION_START_TIME.utcEpochNs` stores LOCAL epoch ns
  despite its name. The two string columns (`utcTime`, `localTime`)
  confirm this: both print the same ISO string (the local wall clock),
  even though the pair is designed to differ. Long-standing misnomer
  across several NSight Systems releases.

  On dual-boot Windows/Linux machines a second effect compounds the
  problem: Windows writes the RTC as LOCAL by default while Linux
  expects UTC, so the hardware clock itself drifts by one UTC-offset
  between boots. If the misalignment isn't exactly one UTC-offset,
  suspect dual-boot RTC too — `timedatectl set-local-rtc 0` fixes the
  Linux side permanently.

Mitigation in FISH code:
  postprocess/analyze_process_gpu.py (and any future per-sqlite
  analyser) derives the true UTC epoch by treating the sqlite's
  utcTime ISO string as local wall-clock and converting via the
  system timezone (zoneinfo, defaults to Europe/Amsterdam; override
  with `FISH_TZ` env var). Do NOT trust the raw utcEpochNs integer.

User action:
  - If local zone is UTC: the bug cancels out and nothing is required.
  - Otherwise: set `FISH_TZ` in the environment (defaults to Europe/
    Amsterdam in this repo) OR accept the default.
  - Debugging: compute
      delta = nsys_utcEpochNs - mongo_first_callback_start_ns
    a value equal to the system UTC offset (in ns) confirms this issue.
  - Longer term: re-ingest nsys data through a fixed timestamp
    normaliser in postprocess/ingest.py::_ingest_nsys. Tracked in
    notes/follow_up.txt (new item — to be added).


ISSUE-003: --cuda-event-trace was OFF → no event↔waitEvent correlation
-----------------------------------------------------------------------
Date discovered: 2026-04-20
Session: fish_20260419_211621 + fish_compose_20260419_161633

Symptom:
  CUPTI_ACTIVITY_KIND_SYNCHRONIZATION shows many type-2 StreamWaitEvent
  rows (pointcloud: 3776, yolo: 3938) but the corresponding
  cudaEventRecord / cuEventRecord calls are almost absent from
  CUPTI_ACTIVITY_KIND_RUNTIME (pointcloud: 0, yolo: 1). Every SYNC
  row's eventId column is -1 (not populated). We initially attributed
  this to "TRT hides its internal event recording from CUPTI" and
  resorted to a time-proximity heuristic for src→dst stream DAG edges
  (notes/gpu_mapping_datapoints.txt obs F, notes/stream_bridge.txt).

Actual root cause:
  Our nsys invocation never enabled --cuda-event-trace. From NVIDIA
  docs: this flag "Trace CUDA Event completion on the device side, and
  get better correlation support among CUDA Event APIs (cudaEventRecord,
  cudaStreamWaitEvent etc.)". Default is OFF in recent nsys versions.

  So the missing records are a configuration choice on our side, not a
  fundamental toolchain limit. With the flag enabled we expect:
    - each type-2 SYNC row to carry a valid eventId
    - matching cudaEventRecord/cuEventRecord calls in RUNTIME
    - deterministic src_stream → dst_stream DAG edges (no heuristic)

Fix applied (2026-04-20):
  - fish_settings.ini [nsys] section: added `cuda_event_trace = true`
  - postprocess-side build_nsys_command reads the setting and adds
    `--cuda-event-trace=true` to the nsys command line (python/fish/gpu.py).
  - launch_wrap (the composable-node path) reads env var
    FISH_CUDA_EVENT_TRACE=1 to enable — off by default because the flag
    may interact badly with multi-threaded component containers on
    older nsys. Enable per run via `export FISH_CUDA_EVENT_TRACE=1`.

Side-effects (per NVIDIA docs):
  - "may increase runtime overhead and the likelihood of false
    dependencies across CUDA Streams" (when cudaEvent timing is not
    disabled). Paper measurement runs accept this cost in exchange for
    deterministic DAG edges.
  - auto mode turns the trace OFF automatically if the target process
    has CUDA_DEVICE_MAX_CONNECTIONS=1.

User action:
  - Existing sessions (pre-2026-04-20) keep the time-proximity
    heuristic. Not re-ingestable — the data wasn't captured.
  - New sessions will have deterministic DAG edges. Expect the
    stream_dag_unresolved field in gpu_task_graph meta to drop to 0.
  - If runs crash under the flag on your CUDA/nsys version, override
    in fish_settings.ini → `cuda_event_trace = false` and fall back
    to the heuristic.
