2026-04-19 (late evening) — oort/oore vertex types for non-executor GPU threads
--------------------------------------------------------------------------------

Investigating yolo_py in the AAS trace to start the GPU-DAG pipeline,
discovered it does not use any ROS 2 callback for inference:

  pid 1252 main thread:      1,029,170 cuda_runtime events, 0 callback_start
  pid 1252 ros_spin_thread:       0 cuda_runtime events, 59,407 callback_start
                                     (all TimeSource.clock_callback — housekeeping)

Source inspection confirms: yolo_node.py's run_inference_loop() runs on
the main thread with a plain while-loop over GStreamer frames + ONNX
Runtime (CUDA). `ros_spin_thread` runs rclpy.spin_once only to keep
sim-time current. Publishing is done via rclpy.Publisher (thread-safe,
no executor required).

Implication for the FISH task model: the GPU work has no F vertex to
attach to, because no callback fires for it. Instead of forcing a
callback-anchored model, we add two new attribute values:

  E.etype = "oore"  (out-of-ROS entity) placeholder — no real ROS
                    subscription/service/timer; exists so the oort
                    has a proper level-2 parent and the FISH
                    hierarchy invariant holds.
  F.ptype = "oort"  (out-of-ROS thread) the non-executor thread that
                    does the real work. gpu_node=True when it issues
                    CUDA calls.

New phase `detect_oort_threads` in model_improved.py, hooked after
attach_callback_groups. Algorithm:

  For each GPU-classified pid (nsys_profiled_pids):
    cuda_tids  = SELECT DISTINCT tid FROM fish.cuda_runtime
                 WHERE session=... AND container=...
    cb_tids    = set of vtids firing callback_start for pid in ros2_trace
    oort_tids  = cuda_tids - cb_tids       # CUDA active, no callbacks
    for each oort tid: create oore E (level 2) + oort F (level 3)
                       under the process's N vertex.

Verified on fish_compose_20260419_161633 aircraft scope:
  1 oort detected (tid=1252, yolo main thread)
  Composed graph: 131 vertices, 180 edges (was 129/178)
  JSON export preserves ptype/etype/thread_tid/gpu_node attributes

Additional findings & follow-ups captured for the paper:

  notes/EXPECTED_OBSERVED.txt §8 — full story of the non-executor
    GPU-thread pattern with measured numbers and source-code
    attribution.
  notes/EXPECTED_OBSERVED.txt §9 — nsys only records CUDA API call

2. First deploy — LTTng buffer overflow
   fish_compose_20260419_154439 (first rclpy run):
     aircraft fish_executor_add_cbgroup: 103,466 events
     aircraft fish_publish_link:              1 event
   fish_publish_link dropped from 88 (C++-only baseline) to 1 because
   the over-firing add_cbgroup events displaced it in the trace
   buffer. yolo_py alone produced ~25,000 add_cbgroup events from
   rclpy.spin_once's add_node/remove_node cycle.

3. Root cause
   rclcpp's Executor::add_callback_group_to_map is one-shot by
   construction (atomic exchange on an `associated_with_executor_`
   flag). rclpy's Executor.add_node is NOT one-shot — it returns
   True on every call where the node is not currently in the set.
   rclpy.spin_once(node) cycles add_node → spin → remove_node every
   iteration, so each iteration looks like a fresh add.

   The rclpy executor itself is NOT recreated each cycle, even
   though spin_once is a standalone function rather than a method.
   `get_global_executor()` returns a module-scoped singleton; each
   Python process has one long-lived rclpy executor. Confirmed by
   the trace: yolo_py (PID 1252) fired fish_executor_init exactly
   once.

4. Fix — bridge-side dedup
   fish_rclpy_trace._SEEN_EXEC_CG: a process-scoped set of
   (id(executor), id(group), id(node)) triples. trace_executor_add_cbgroup
   short-circuits on repeat triples. See notes/dedup_rclpy_exec.txt
   for design rationale, alternative designs rejected, and an
   audit-health snippet.

5. Second deploy — clean counts
   fish_compose_20260419_161633 (same mission, with dedup):
     aircraft fish_executor_add_cbgroup:  34 events (~3000x reduction)
     aircraft fish_publish_link:          89 events (recovered)
     aircraft fish_executor_init:         26  (13 C++ + 13 rclpy)
     aircraft fish_callback_group_init:   50  (24 C++ + 26 rclpy)
     yolo_py (PID 1252) EX vertex:  SingleThreaded, 1 thread, 1 CG
     9/9 EX vertices in unified graph have executor_type + cb_groups
     (previously 8/9, yolo_py was ?)

6. Paper material
   EXPECTED_OBSERVED.txt now has §7 documenting the rclpy
   add_node/remove_node cycle, the observation-side vs construction-
   side invariant mismatch, and the measured before/after impact.
   dedup_rclpy_exec.txt captures the mechanism in detail.

   The generic lesson for the paper: structurally "the same" API in
   rclcpp vs rclpy can have different invariants when observed at
   runtime. Dedup belongs on the observation side, not baked into
   the tracepoint firing convention.

6. Key findings, paper-relevant
   See notes/EXPECTED_OBSERVED.txt. Summary:
   - Processes have MULTIPLE executors, not one. Two mechanisms:
     transient ST executors from rclcpp::spin() / spin_some() helpers,
     and deliberate persistent parallel executors (e.g., px4_interface
     runs one 20-thread MT + one ST side-by-side).
   - Nodes have MULTIPLE callback groups with MIXED types. AAS aircraft
     has 14 MutuallyExclusive + 23 Reentrant CGs. PX4 /fmu/out/* sensor
     topics specifically use Reentrant so messages can be dispatched
     concurrently on different MT-executor threads.
   - MultiThreadedExecutor default num_threads is
     `std::thread::hardware_concurrency()`, i.e., 20 on this host. Not
     specified anywhere in the AAS source. Same binary on a 4-core
     machine would use 4 threads — scheduling analysis done offline
     from source is structurally blind to this.
   - The canonical ROS 2 scheduling model (1 process = 1 executor,
     1 node = 1 CG) does not match any of the 3 AAS containers.

rclpy note (don't assume the Python side silently works)
---------------------------------------------------------

The yolo_py process (the only rclpy-based node in this mission)
produced ZERO scheduler events. I concluded "rclpy doesn't have
scheduler tracepoints", which is TRUE for the scheduler-specific
tracepoints (install_rclpy.sh / fish_rclpy_trace.py do NOT yet patch
rclpy/executors.py or its CallbackGroup classes), but the observation
is also muddied by yolo_py's own rough start:

  - PID 712 ran normally: 1 rcl_node_init, 12,374 callback_start
  - GPU daemon killed it to relaunch under nsys → PID 1228
  - PID 1228 never initialized (0 rcl_node_init, 0 callback_start) —
    silent crash during Python/CUDA startup under nsys
  - Eventually recovered as PID 1249: 1 rcl_node_init,
    40,718 callback_start (rclpy subscription / timer / service
    tracepoints were firing normally here)

So the existing rclpy tracepoints (rclpy_*_callback_added,
rclpy_callback_register) ARE working — we have PID-1249 callbacks and
subscription links. What is MISSING is the rclpy-specific
counterparts for the 4 new scheduler tracepoints. Adding them is
mechanically straightforward: patch rclpy/executors.py
SingleThreadedExecutor.__init__, MultiThreadedExecutor.__init__,
MutuallyExclusiveCallbackGroup.__init__, ReentrantCallbackGroup.__init__,
Executor.add_node to call via the existing fish_rclpy_trace ctypes
bridge into the tracetools library.

Also worth calling out for the paper: rclpy's MT executor runs under
the GIL, so concurrency behavior is materially different from rclcpp
MT even with the same CG types. When the rclpy scheduler patch is
added, an additional `lang: "python"` flag on the EX vertex would let
analysis correctly apply GIL constraints.
