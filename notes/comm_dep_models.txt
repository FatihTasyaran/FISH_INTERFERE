FISH Communication Patterns — Atomic Definitions
===================================================

Terminology
-----------
  Entity:   A node's callback implementation. Entity type = callback type.
            Types: sub, serv, tmr (only these three)
            An entity may have additional aspects (pub, sub, cli) that
            determine its participation in communication patterns.

  Aspect:   A feature of an entity within a communication pattern.
            - pub:  entity's callback publishes on a topic
            - sub:  entity's callback is triggered by a topic message
            - cli:  entity's callback calls a service (request/response)
            Aspects determine edge direction. They are NOT entity types.

  Aspect list:  Each entity carries a list of aspects describing all
                communications it participates in. Examples:
                  (aspect: sub, topic: /lidar_points, msg: PointCloud2)
                  (aspect: pub, topic: /kiss/odometry, msg: Odometry)
                  (aspect: cli, service: /some_service)


==========================================================================
1. Publisher-Subscriber (via Topics)
==========================================================================

Elements:
  - Publisher:   an entity whose callback publishes a message on a topic
  - Subscriber:  an entity whose callback receives a message from a topic
  - Topic:       named channel with a message type
  - Message:     the data transferred

Any entity type (sub, serv, tmr, cli) can have a publisher aspect.
The subscriber side is always an entity with etype="sub".

Edge:
  publishing_entity ════> subscribing_entity
  nature = "msg"

Aspect examples:
  Entity: [SUB] /lidar_points on kiss_icp_node
    aspects:
      - aspect: sub,  topic: /lidar_points,  msg: PointCloud2
      - aspect: pub,  topic: /kiss/odometry, msg: Odometry
      - aspect: pub,  topic: /tf,            msg: TFMessage
      - aspect: pub,  topic: /kiss/frame,    msg: PointCloud2
      - aspect: pub,  topic: /kiss/keypoints,msg: PointCloud2
      - aspect: pub,  topic: /kiss/local_map,msg: PointCloud2

  Entity: [TMR] timer_20ms on px4_offboard
    aspects:
      - aspect: pub,  topic: /Drone1/fmu/in/trajectory_setpoint, msg: ...

ASCII representation:

              Node A                              Node B
           ┌──────────┐                       ┌──────────┐
           │          │                       │          │
           │  [SUB]   │     /topic_X          │          │

Attribution method:
  For each rcl_publish event, find the enclosing callback_start/end
  window on the same (vpid, vtid). The callback's entity gains a
  publisher aspect for that topic.

  Match: (vpid, vtid, callback_start.ts < rcl_publish.ts < callback_end.ts)
  Resolve: rcl_publish.publisher_handle → rcl_publisher_init → topic_name

==========================================================================
2. Service (Client-Server)
==========================================================================

Elements:
  - Client:    an entity that sends a request (etype="cli")
  - Server:    an entity that receives a request and sends a response (etype="serv")
  - Service:   named endpoint with request/response types
  - Request:   data sent client → server
  - Response:  data sent server → client

Edge:
  client_entity ────> server_entity
  nature = "dep" (dependency — client depends on server's response)

Aspect mapping:
  Client entity aspects:
    - aspect: req_pub,  service: /srv_name  (sends request)
    - aspect: res_sub,  service: /srv_name  (receives response)

  Server entity aspects:
    - aspect: req_sub,  service: /srv_name  (receives request)
    - aspect: res_pub,  service: /srv_name  (sends response)

  Unlike pub/sub, service aspects are always paired — a client always
  has both req_pub + res_sub, a server always has req_sub + res_pub.

ASCII representation:

              Node A                              Node B
           ┌──────────┐                       ┌──────────┐
           │          │                       │          │
           │  [CLI]   │     /service_name     │  [SRV]   │
           │  etype=  │ ─────────────────>    │  etype=  │
           │  "cli"   │   request (req)       │  "serv"  │
           │          │                       │    │     │
           │          │                       │    v     │
           │          │                       │  [F] cb  │
           │          │                       │    │     │
           │          │     /service_name     │    v     │
           │          │ <─ ─ ─ ─ ─ ─ ─ ─ ─   │ response │
           │          │   response (res)      │          │
           └──────────┘                       └──────────┘

  Edge direction is client → server (dependency direction).
  The response flows back implicitly — not a separate edge.

Structural discovery:
  - Client: rcl_client_init → client_handle, node_handle, service_name
  - Server: rcl_service_init → service_handle, node_handle, service_name
  - Match key: service_name (must be identical)

  No runtime service-call event exists in ros2_tracing. The DDS layer
  handles request/response as rmw_publish/rmw_take but the service
  name is lost at that level. Only structural init events available.

  rmw_service_handle and rmw_client_handle are present in trace but
  do NOT appear in runtime rmw_take/rmw_publish events — these handles
  are DDS-internal. rmw_take uses rmw_subscription_handle which is a
  different object. Cross-process message pointer matching also does not
  work (DDS serialize/deserialize creates new pointers per process).

  Consequence: service call runtime timing (request sent → response
  received) cannot be directly measured from current trace events.
  Only structural connectivity is available. For runtime timing,
  custom tracepoints (rmw_send_request/rmw_take_response) would be
  needed — this is a potential future FISH tracepoint extension.

  Note on rclpy action clients (e.g. ros2 action send_goal CLI):
    - callback_start/end does NOT fire for these nodes. The CLI tool
      uses its own event loop, not the patched rclpy executor.
    - This does not affect structural edge construction (rcl_client_init
      is sufficient) but means no runtime callback timing for CLI clients.


Service client as entity aspect (not entity type):

  Client is NOT an entity type. It is an ASPECT of another entity.
  An entity (sub, serv, tmr) whose callback calls async_send_request()
  has a "client aspect" for that service.

  Why not entity type:
    - Client callback is OPTIONAL (future-only vs callback overload)
    - Client callback is registered per-request, not at init time
    - rclpy has NO client callback at all (future-only)
    - ROS 2 tracing has ZERO tracepoints for client (no callback_start/end,
      no callback_register, no executor_execute)
    - See client_implementation.txt for full source analysis

  The client callback (when present) is a "continuation" of the
  requesting entity's work — executed later by the same executor
  when the response arrives.

  Service call model with client aspect:

       Entity [TMR] timer_cb (Node A)
       │
       │ callback_start
       │   work...
       │   async_send_request(req) ───────────────────┐
       │   work...                                     │ request
       │ callback_end                                  │ (DDS)
       │                                               ▼
       │                                    Entity [SRV] server_cb (Node B)
       │                                    │ callback_start
       │                                    │   process request
       │                                    │   send response
       │                                    │ callback_end
       │                                               │
       │              ┌────────────────────────────────┘
       │              │ response (DDS)
       │              ▼
       │ handle_response (Node A, same executor)
       │   continuation:
       │     promise.set_value(response)
       │     or: callback_start → callback(future) → callback_end
       │
       │ ◄── round-trip delay measurable ──►
       │ (request_sent.ts → response_received.ts)

  FISH custom tracepoints (rclcpp client.hpp patch):

    1. fish_rclcpp_client_request_sent(client_handle, sequence_number)
       When: async_send_request_impl, after rcl_send_request succeeds
       Links: client_handle → rcl_client_init → service_name

    2. fish_rclcpp_client_response_received(client_handle, sequence_number, response_type)
       When: handle_response entry, before variant dispatch
       response_type: 0=future_only, 1=callback, 2=callback_with_request
       Links: sequence_number matches request_sent, measures round-trip

    3. callback_start/end (reused existing tracepoint)
       When: around callback invocation in handle_response (type 1 and 2 only)
       Only fires when callback variant is used, not for future-only

  These tracepoints make the previously invisible execute_client path
  fully observable in trace data.


==========================================================================
3. Action (Composite Pattern)
==========================================================================

An action is a composite communication pattern built from patterns 1
and 2. It decomposes into 3 service pairs + 2 topic pairs.

Elements:
  - Action server:  a node that accepts goals and produces results
  - Action client:  a node (or entity with cli aspect) that sends goals
  - Goal:           the requested task (send_goal service)
  - Result:         the final outcome (get_result service)
  - Feedback:       periodic progress updates (topic)
  - Cancel:         cancellation request (cancel_goal service)
  - Status:         goal status updates (topic)

Decomposition:
  Action = 3 × Pattern 2 (service) + 2 × Pattern 1 (pub/sub)

  Server node components:
    [SRV] /_action/send_goal      server entity, callback: goal handler
    [SRV] /_action/cancel_goal    server entity, callback: cancel handler
    [SRV] /_action/get_result     server entity, callback: result handler
    PUB   /_action/feedback       publisher aspect of a server entity
    PUB   /_action/status         publisher aspect of a server entity

  Client node components:
    cli aspect: /_action/send_goal      on the requesting entity
    cli aspect: /_action/cancel_goal    on the requesting entity
    cli aspect: /_action/get_result     on the requesting entity
    [SUB] /_action/feedback             subscriber entity
    [SUB] /_action/status               subscriber entity

  Abstract grouping:
    [ACT] action entity (not a callback, groups the above components)

Collapsed view (action as single entity):

       Client Node                            Server Node
    ┌──────────────┐                       ┌──────────────┐
    │              │                       │              │
    │  [entity]    │     action_name       │  [ACT]       │
    │  with cli    │ ──────────────────>   │  abstract    │
    │  aspect      │     (composite)       │  grouping    │
    │              │                       │              │
    └──────────────┘                       └──────────────┘

Expanded view (action decomposed into primitives):

       Client Node                            Server Node
    ┌──────────────┐                       ┌──────────────┐
    │              │                       │              │
    │  [entity]    │  send_goal (svc)      │  [SRV]       │
    │  cli aspect ─┼──────────────────>    │  goal_cb     │
    │              │                       │    └─[F]     │
    │  [entity]    │  cancel_goal (svc)    │  [SRV]       │
    │  cli aspect ─┼──────────────────>    │  cancel_cb   │
    │              │                       │    └─[F]     │
    │  [entity]    │  get_result (svc)     │  [SRV]       │
    │  cli aspect ─┼──────────────────>    │  result_cb   │
    │              │                       │    └─[F]     │
    │              │                       │              │
    │  [SUB]       │  feedback (topic)     │  PUB         │
    │  feedback_cb │ <═════════════════    │  (on a SRV   │
    │              │                       │   entity)    │
    │  [SUB]       │  status (topic)       │  PUB         │

                                           [ACT] abstract
                                           references all 5
                                           server components

Runtime trace sequence (action goal lifecycle):

  Client entity callback (Node A):
    callback_start(entity_cb)
      ...
      fish_rclcpp_client_request_sent(cli_handle, seq=1)     ← send_goal request
    callback_end(entity_cb)

    fish_rclcpp_client_response_received(cli_handle, seq=1)  ← goal accepted
    fish_rclcpp_client_request_sent(cli_handle, seq=1)       ← get_result request

  Server (Node B):
    fish_rclcpp_action_execute_goal(action_handle)            ← goal dispatch
      callback_start(goal_srv_cb)
        ... process goal ...
        rcl_publish(feedback_handle)                     ← feedback published
      callback_end(goal_srv_cb)

    fish_rclcpp_action_execute_result(action_handle)          ← result dispatch
      callback_start(result_srv_cb)
        ... prepare result ...
      callback_end(result_srv_cb)

  Client (Node A):
    fish_rclcpp_client_response_received(cli_handle, seq=1)  ← result received

  Timing measurements:
    Goal accept latency:   send_goal request_sent → response_received
    Execution time:        execute_goal → execute_result
    Result latency:        get_result request_sent → response_received
    Total action duration: first request_sent → last response_received
    Feedback period:       successive rcl_publish on feedback topic

Edge summary per action:
  3 service edges (Pattern 2):  client entity → [SRV] send_goal
                                client entity → [SRV] cancel_goal
                                client entity → [SRV] get_result
  2 topic edges (Pattern 1):   [SRV] entity (pub aspect) → [SUB] feedback
                                [SRV] entity (pub aspect) → [SUB] status

Trace events used:
  Structural:
    fish_rclcpp_action_server_init    → action_handle, service handles
    rcl_client_init              → client service names
    rcl_service_init             → server service names
    rcl_subscription_init        → feedback/status subscribers
    rcl_publisher_init           → feedback/status publishers

  Runtime:
    fish_rclcpp_action_execute_goal   → goal dispatch (FISH custom)
    fish_rclcpp_action_execute_cancel → cancel dispatch (FISH custom)
    fish_rclcpp_action_execute_result → result dispatch (FISH custom)
    fish_rclcpp_client_request_sent   → request timing (FISH custom)
    fish_rclcpp_client_response_received → response timing (FISH custom)
    callback_start/end           → callback execution timing
    rcl_publish                  → feedback/status publish timing

Verified: session fish_20260330_234647, 3 actions (takeoff, orbit, land).
  request_sent + response_received pairs observed for all 3 actions.
  Goal accept latency ~1ms, result latency 6-10s (flight maneuver time).
