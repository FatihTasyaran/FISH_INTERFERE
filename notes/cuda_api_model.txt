CUDA API → FISH GPU DAG model mapping
======================================

Purpose:
  Document every CUDA API call we encounter in traces and how it maps
  into the FISH Typed DAG model (PDF §4.5: CPU / SM / CE nodes).

  Some calls become graph nodes, some create edges, some are init-only
  bookkeeping, some are pure noise that should be filtered. This table
  is the contract between `postprocess/gpu_dag.py` and the trace data.

Source session for numbers: fish_compose_20260419_161633 (aircraft, tid=1252,
yolo_py with ONNX Runtime CUDAExecutionProvider).


Classification legend
---------------------

  PHASE     Init    fires once per process (skip from steady-state DAGs)
            Steady  repeats per-invocation (contributes to DAG)
            Tail    shutdown / cleanup (exclude from steady-state)

  NODE      CPU     host-side driver call, becomes a CPU DAG vertex
            CPU+SM  host launch → GPU kernel on an EE (compute engine)
            CPU+CE  host launch → GPU memory transaction on a copy engine
            --      no DAG node generated (filtered or metadata)

  BOUNDARY  X       candidate period boundary when followed by host idle
            --      not a boundary

  NOTES     anything special that affects DAG construction

========================================================================

API                               PHASE  NODE  BOUNDARY  NOTES
-------------------------------   -----  ----  --------  -------------------

                                                         Skip. Should be the

                                                         Skip.

cuLibraryLoadFromFile /                                  GPU. One-time per

                                                         Skip when isolated,

                                                         below for steady-state
                                                         use.

cudaMalloc* /                     Init   CPU   --        Persistent buffers
cudaMallocAsync /                                        allocated once. If
cudaMallocHost                                           observed in steady
                                                         state → dynamic growth
                                                         warning (usually a
                                                         performance bug, not
                                                         intentional).

cudaStreamCreate* /               Init   --    --        Streams / events are
cudaEventCreate*                                         allocated once and
                                                         reused for the process
                                                         lifetime.

cuGetProcAddress_v2               Init   --    --        Driver function
                                                         resolution (some
                                                         runtime libraries call
                                                         this on first use).
                                                         Skip.

cuModuleGetLoadingMode            Init   --    --        One-time driver query.

Detection heuristic for the init boundary:
  T_init_end = max(
    last_ts(cuLibraryLoadData, cuModuleLoad*, cudaMalloc,
            cudaStreamCreate*, cudaEventCreate*)
  )
  + small cooldown (e.g. 10ms for driver settling).

  Steady-state period mining happens ONLY on calls with
  ts > T_init_end.

SYNCHRONIZATION — period boundary candidates
========================================================================

API                               PHASE   NODE  BOUNDARY  NOTES
-------------------------------   -----   ----  --------  --------------------
cudaStreamSynchronize*            Steady  CPU   X         Host blocks until
                                                          stream's work
                                                          completes. If followed
                                                          by > gap_ms host idle
                                                          → period boundary
                                                          (see gpu_dag.py).
                                                          In yolo: 8,076
                                                          occurrences, 1,962
                                                          of them followed by
                                                          > 5ms idle (real
                                                          period ends).

cudaDeviceSynchronize*            Steady  CPU   X         Host blocks until ALL
                                                          work on the device
                                                          completes. Stronger
                                                          boundary than
                                                          stream-sync. Same
                                                          treatment.

cudaEventSynchronize*             Steady  CPU   Maybe     Host blocks until
                                                          event triggers.
                                                          Boundary only if
                                                          followed by idle;
                                                          often used mid-period
                                                          for internal staging,
                                                          not period end.

cudaStreamWaitEvent*              Steady  CPU   --        Cross-stream
                                                          dependency: stream B
                                                          waits on event from
                                                          stream A. NOT a
                                                          host-side block.
                                                          Introduces a CE→SM
                                                          cross-stream DAG edge
                                                          that we do not yet
                                                          model (single-stream
                                                          yolo doesn't need it).
                                                          For multi-stream
                                                          apps (Autoware
                                                          perception, custom
                                                          CUDA code): FUTURE
                                                          work.

cudaEventRecord*                  Steady  CPU   --        Places an event
                                                          marker in a stream.
                                                          Pure CPU enqueue;
                                                          the event fires later
                                                          when preceding stream
                                                          work completes.
                                                          Relevant for
                                                          cross-stream edges
                                                          (paired with
                                                          cudaStreamWaitEvent).

Summary of DAG-producing ops in yolo_py (fish_compose_20260419_161633)
========================================================================

Dominant DAG (count=1,857 / 1,863):
    1050 total nodes
      550 CPU    — host-side driver calls
      458 SM     — kernel launches via cudaLaunchKernel / cuLaunchKernel
       42 CE     — cudaMemcpyAsync → H2D + D2H

Per-period API breakdown (approximate, divide total by 1,863 dominant-shape
invocations):

     kernel launches     ~460    → 458 SM nodes + 2 outliers
     memcpyAsync         ~ 42    →  42 CE nodes
     memset              ~  9    →  ~9 CPU nodes (no CE match yet; GAP)
     kernel-get-function ~ 36    →  CPU nodes, tied to cuLaunchKernel
     streamSynchronize   ~  4    →  CPU nodes, one is the period boundary
     launchHostFunc      ~  1    →  CPU node

     ≈ 550 CPU nodes per period — accounts for the full count.

The asymmetry between 550 CPU and 458 SM is NOT pathological — it's
expected because (a) every kernel launch also has a runtime-API sibling
that CUPTI logs, (b) memsets and host-func enqueues add CPU-only
nodes, (c) sync and get-function calls are CPU-only.


========================================================================
Known gaps / TODOs in the current gpu_dag.py
========================================================================

  G1. cudaMemset — not ingested; produces CPU-only DAG nodes. Fix:
      extend ingest.py to pull CUPTI_ACTIVITY_KIND_MEMSET into a new
      `gpu_memset` measurement with session/container/correlationId/
      streamId/bytes/duration_ns. Then gpu_dag.py builds CE nodes for
      these too.

  G2. Sync memcpy (non-async) — current code emits CPU+CE but should
      emit 2×CPU bracketing the CE per PDF §4.7. yolo has 208 sync
      memcpys out of 78k memcpys (0.3%), negligible for yolo — but
      sync-memcpy-heavy apps (pre-ORT CUDA) will miss the host-block
      semantics.

  G3. Cross-stream event edges — cudaEventRecord + cudaStreamWaitEvent
      pairs define explicit cross-stream dependencies. Today we emit
      only within-stream FIFO edges. For multi-stream apps this is a
      real gap. yolo is effectively single-stream so unaffected.

  G4. Error-status noise — cudaGetLastError calls are not filtered.
      yolo doesn't call them; other apps will produce inflated CPU
      node counts. Easy filter at gpu_dag.build_typed_dag.

  G5. Init / tail phase detection — currently gpu_dag processes
      every CUDA call after the first sync boundary. Init-phase
      calls (module load, first cudaMalloc) are swept up into the
      first few "periods" and appear as 1-count outlier shapes
      (shapes [2]-[5] in our run). Implementing the T_init_end
      heuristic above would cleanly separate init from steady
      state.

Empirical per-shape analysis from yolo_py
(session fish_compose_20260419_161633, aircraft, tid=1252)
========================================================================

After sync-pattern period detection + min_gpu_ops=2 filter we get
1,863 productive invocations distributed across exactly 6 unique
DAG signatures. Every one of those signatures has a clean semantic
explanation — the process's ENTIRE CUDA lifetime is covered by 6

  [0]  CANONICAL INFERENCE             — count=1,857 (99.7%)

  Signature:        45f12736fc58daf2
  Nodes:            1,056  (CPU=550, SM=464, CE=42)
  Edges:            1,560
  Stream:           single (29)

  [2]  INIT PHASE (process startup)    — count=1  (once per process)
  ------------------------------------------------------------------

  Signature:        9c0b1bfa7cd8b3e8
  Nodes:            2,153  (CPU=1,944, SM=1, CE=208)
  Edges:            2,359
  Top ops:
    cuGetProcAddress_v2                 × 1,257  (!)
    cudaMemcpy_v3020  (synchronous)     ×   208
    memcpy_H2D                          ×   208
    cudaStreamSynchronize               ×   208
    cudaStreamSynchronize_v3020         ×   208
    cudaEventCreateWithFlags_v3020      ×    19
    cudaStreamCreateWithFlags_v5000     ×    16
    cudaMalloc_v3020                    ×     8
    cuInit                              ×     5
    cuModuleGetLoadingMode              ×     4
    cudaGetDeviceProperties_v12000      ×     2   (2.8 ms each — expensive!)
    cudaFree_v3020                      ×     2

  Interpretation: the FULL "boot the CUDA world" sequence:

    1. Driver bootstrap.
       cuInit ×5 + cuGetProcAddress_v2 ×1,257 — libcuda.so's dynamic
       symbol table is being walked to resolve every entry point the
       ORT CUDA provider will ever call.  This cost is amortized
       because it happens ONCE.

    2. Capability discovery.
       cudaGetDeviceProperties + cudaMemGetInfo + cuModuleGetLoadingMode.
       2.8 ms per property query — these are NOT free.

    3. Resource pool allocation.
       16 streams + 19 events + 8 persistent allocations (cudaMalloc).
       Steady state uses only stream 29 for compute; the other 15
       are ORT-internal scratch streams that mostly sit idle.

    4. Model weight upload — 208 synchronous H2D transfers.
       For yolo26n every weight tensor is moved host→device one by
       one with `cudaMemcpy_v3020` (blocking), surrounded by pre/post
       `cudaStreamSynchronize` pairs. This is EXACTLY the PDF §4.7
       "sync memcpy = 2×CPU + 1×CE bracket" pattern, observed in the
       wild 208 times in a single init phase.

    5. ONE kernel executes.
       Almost certainly a cuBLAS / cuDNN library-probe kernel (these
       libraries benchmark candidate algorithms on first use to pick
       the fastest for the target GPU).

  Detector implication (G5 — init-phase detector).
  If a period contains ≥5 `cuInit` OR ≥100 `cuGetProcAddress_v2` OR
  ≥5 `cudaMalloc`, classify it as the init phase and exclude from
  steady-state DAG aggregation. yolo's init burst contains all three
  flags; a sharper threshold based on just one of them is sufficient.


  [5]  END PHASE (truncated last inference)  — count=1
  -----------------------------------------------------

  Signature:        3a63b3495a4ca84d
  Nodes:            718  (CPU=373, SM=328, CE=17)
  Edges:            (~0.68 × canonical)
  Content: first 15 ops are IDENTICAL to the canonical DAG — opens
  with the H2D input memcpy, implicit_convolve_sgemm, generic tensor
  ops, unary elementwise… Then cuts off partway through. Roughly
  two-thirds of a yolo26n forward pass.

  Interpretation: process was killed mid-inference. fish_orchestrator
  calls `ros2 run fish stop` on the aircraft container, which in turn
  sends SIGTERM to yolo_py. The signal arrived while the main loop
  was in the middle of `do_yolo(frame)` — `cudaLaunchKernel` returned
  but nsys's buffer flush captured only the ops that had reached it
  before the process died. Whatever was mid-flight on the GPU was
  abandoned (no `cudaDeviceReset`, no `cudaFree`).

  ADDITIONALLY — the actual final CUDA events on tid=1252 (after this
  truncated DAG was filtered as its own period) are 20 consecutive
  `cudaStreamSynchronize` pairs with 30-60ms gaps between them:

      [1029130] +43ms   cudaStreamSynchronize  ← large gap
      [1029131] +0.1ms  cudaStreamSynchronize  ← ORT's paired ack
      [1029132] +38ms   cudaStreamSynchronize  ← next frame-wait
      [1029133] +0.1ms  cudaStreamSynchronize
      …
      [1029169] +0.1ms  cudaStreamSynchronize  — last ever CUDA call

  This is ORT's "wait for next frame" idle loop — the process already
  finished its mid-inference truncated DAG, then was polling on
  streams waiting for more work. SIGTERM fired during that poll.

  Detector implication.
  There is no cleanup DAG to detect on shutdown — real ROS 2
  applications do NOT call `cudaDeviceReset` or `cudaFree` before
  process exit. Kernel / driver cleanup is left to process teardown
  semantics. Worth flagging in the paper: CUDA state at SIGTERM is
  left dirty, GPU ownership released via /proc/<pid> teardown, not
  via CUDA API.


  [1]  DOUBLE-FRAME COALESCE          — count=2
  [4]  TRIPLE-FRAME COALESCE          — count=1
  ----------------------------------------------

  [1] signature:    3c90e07189ef8001  nodes=2,112 = 2×1,056 canonical
  [4] signature:    4357635d5ee25c34  nodes=3,168 = 3×1,056 canonical

  Interpretation: two (or three) consecutive inference invocations
  with so little idle between them that our 5ms gap-after-sync period
  boundary never triggered. The content is EXACTLY the canonical DAG
  repeated back-to-back.

  These are not a "new kind of work" — they are steady-state
  invocations that the detector failed to split. Root cause: during
  bursty moments (network glitch + frame catch-up), frames arrive
  within < 5ms of each other, yolo's main loop proceeds without
  blocking on `frame_queue.get()`, and there is no host-idle window
  between the two inference cycles.

  In aggregate: 2×[1] + 3×[4] = 7 "lost" inferences that are
  attributed to the wrong DAG. On a sample of 1,870 inferences,
  that's 0.37 % loss — acceptable but worth quoting.

  Detector improvements that would eliminate these:
    (a) Raise gap threshold — but this merges genuine 3-4ms idle
        periods with steady state. No clean tuning.
    (b) Detect the repeating pattern SIGNATURE and split when two
        (identical, full-shape) DAGs appear concatenated.
    (c) Use NVTX ranges if the app emits them (yolo/ORT does not).
    (d) Explicit periodicity anchor — look for the specific
        cudaMemcpyAsync→H2D→implicit_convolve_sgemm opening signature
        and split on every recurrence. This is application-specific
        but catches the case cleanly.

  For yolo (0.37 % error) we leave as-is. For future apps with more
  frequent bursts, (d) is the cleanest fix.


  [3]  WARMUP / TAIL MINOR VARIATION   — count=1
  -----------------------------------------------

  Signature:        d76ae601387f1415
  Nodes:            1,962  (CPU=1,455, SM=465, CE=42)
  Content: top frequency dominated by `cuGetProcAddress_v2 ×838` —

========================================================================
What this tells us about real-world GPU apps for the paper
========================================================================

(Also tracked in notes/paper_to_do.txt.)

1. Real ROS 2 GPU applications have a rich lifecycle — init,
   steady-state, end — and the TASK model has to distinguish them.
   Analyses that assume "a GPU task is a periodic DAG with fixed
   shape" are correct for 99.7 % of invocations but miss the 0.3 %
   that shape the mission's startup + shutdown latency.

2. The sync-memcpy "2×CPU + 1×CE" bracket from PDF §4.7 IS the real
   pattern used by production code (observed 208 times in yolo init
   alone).

3. SIGTERM-driven shutdown leaves GPU state dirty. Real systems do
   NOT call cleanup API. CUDA context teardown happens via process
   exit, which is invisible to our profiling. Any analysis that
   models "process termination = cudaDeviceReset" is wrong.

4. Short idle periods (< 5ms) can coalesce consecutive invocations
   even under pattern-based detection. An app-specific periodicity
   anchor (e.g., look for the recurring opening sub-sequence of the
   canonical DAG) is a robust fallback.

5. The init phase alone is a distinct, identifiable DAG — not a
   noise period. `cuGetProcAddress_v2` × 1,000+ is a clear
   fingerprint of dynamic symbol loading; `cudaMemcpy` × 208 with
   sync pairs is a clear fingerprint of model-weight upload. The
   init DAG's own structure tells us what ORT spent its startup
   budget on (answer: mostly symbol resolution + weight upload,
   actual kernel compile is negligible).

Universal-app detector outline (next iteration of gpu_dag.py)
========================================================================

```
1. Load all CUDA API calls for (session, container, tid).

2. Skip init:
     T_init_end = max_ts_of(cuLibraryLoadData, cuModuleLoad*,
                            cudaMalloc, cudaStreamCreate*,
                            cudaEventCreate*) + 10ms
     filter calls to ts > T_init_end

3. Filter noise:
     drop rows with api_name ∈ {cudaGetLastError, cudaPeekAtLastError,
                                cudaGetDevice, cudaSetDevice,
                                cuCtxSetCurrent, cuCtxGetCurrent,
                                cuStreamSetAttribute*, cuGetProcAddress_v2,
                                cuModuleGetLoadingMode}

4. Choose period boundary source (priority):
     if NVTX ranges present with non-trivial text: use them
     elif cudaStreamSynchronize* / cudaDeviceSynchronize*
          with > gap_ms idle after: use them (current)
     else: n-gram repetition on api_name sequence

5. Separate warm-up:
     first K periods (K≈5 default, tunable): mark as warm-up, keep
     in a separate `gpu_dags_warmup` attribute for JIT / cache
     population baseline.

6. Build DAG per-invocation, canonicalize signature.

7. Aggregate unique shapes by count; report dominant + tail.
```
