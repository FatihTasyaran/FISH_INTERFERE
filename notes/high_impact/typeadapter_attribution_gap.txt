# HIGH-IMPACT: rclcpp::TypeAdapter callback-attribution gap in FISH
# Date: 2026-06-08
# Why this matters: this is the first known case where FISH's "every
# rclcpp subscription is fully attributed" property does NOT hold.

## TL;DR

FISH's E→F (callback) attribution depends on the `ros2:rclcpp_subscription_callback_added` upstream tracepoint firing. We observed
this tracepoint fire for 19 of 20 subs in the apriltag pipeline. The one
subscription that does NOT fire it uses `rclcpp::TypeAdapter` template
specialization in its construction path. That means: any ROS 2 application
whose user code subscribes via TypeAdapter to a topic where the peer
publisher does NOT also use TypeAdapter currently slips past FISH's
attribution layer.

For the FISH paper, this softens the "FISH covers all rclcpp callback
dispatch" claim. The framework still captures the E (rcl_subscription_init
fires) — only the F-binding is missed. Pipeline-level coverage of E
vertices remains intact.

## Where we hit it

apriltag pipeline, MonitorNode subscribing to /r2b/apriltag_detections:
  - AprilTagNode publishes via plain rclcpp::create_publisher<AprilTagDetectionArray> (no NITROS-aware peer on the topic)
  - MonitorNode is NitrosMonitorNode → NitrosSubscriber NEGOTIATED →
    createCompatibleSubscriber → type-manager
    createCompatibleSubscriberCallback<NitrosDetection2DArray>
  - That callback ends up instantiating Subscription<rclcpp::TypeAdapter<...>>
  - The TypeAdapter specialization's ctor does not flow through the
    same TRACEPOINT(rclcpp_subscription_init, ...) and
    TRACEPOINT(rclcpp_subscription_callback_added, ...) call sites that
    the non-TypeAdapter Subscription ctor uses

Trace evidence:
  - 19 / 20 subs: rcl_subscription_init + rclcpp_subscription_init +
    rclcpp_subscription_callback_added all fire ✓
  - 1 sub (/r2b/apriltag_detections): only rcl_subscription_init fires ✗

Per-vertex impact in apriltag: 1 missing F out of 79 expected (1.3%).
Per-pipeline impact: 1 E without F child; downstream FISH graph
attribution misses linking the message arrival to the user callback.

## Scope of the limitation

NITROS-specific in the apriltag observation, BUT the underlying mechanism
is generic rclcpp::TypeAdapter behaviour. The set of applications
potentially affected:

a) NITROS pipelines where a NITROS subscriber peers with a non-NITROS
   publisher (cross-stack boundary). Very common in Isaac ROS benchmarks
   where Monitor/Playback hook into user pipelines.

b) Any rclcpp application that defines a custom type via
   rclcpp::TypeAdapter (e.g. converting std::vector to a ROS array) and
   subscribes to a topic published by a non-TypeAdapter peer. This is
   not a mainstream pattern but is in core rclcpp.

c) Mainstream ROS 2 apps that publish/subscribe natively (no TypeAdapter
   anywhere): unaffected. This is most of the ecosystem.

Honest framing for the paper:
  "FISH captures every E vertex in any rclcpp pipeline. Its F-binding
  attribution is complete for any subscription created through the
  standard rclcpp::Subscription path. Subscriptions instantiated via
  rclcpp::TypeAdapter specialization are an attribution edge — the E
  is captured, the F is not, until a NITROS-side tracepoint is added."

## Why we can't audit "every app in the world"

The TypeAdapter mechanism is part of upstream rclcpp and is open to any
package author. Estimating prevalence:
  - Isaac ROS uses it heavily for NITROS message-type abstraction
  - Some ROS 2 community packages register TypeAdapters for plain C++
    types to avoid serialization overhead
  - The pattern is mostly in performance-sensitive subsystems, not
    everyday business-logic nodes

We cannot guarantee zero TypeAdapter usage across every ROS 2 package
ever written. We can guarantee:
  1. The E (rcl_subscription_init) is captured for all of them.
  2. The F-binding gap, when present, is bounded to "one F per
     TypeAdapter-specialised subscription".
  3. Adding a single NITROS-side tracepoint (see
     apriltag_model_validation.txt § Open follow-ups) closes the gap
     for the NITROS case completely.
  4. A more general fix would be an rclcpp-side tracepoint at the
     TypeAdapter specialization's subscription-construction site,
     applicable to any TypeAdapter user. This is the upstream-fix
     option.

## Paper position

Recommended caveat in the FISH paper § coverage section:

  "FISH provides complete attribution for the standard rclcpp + rclpy
  callback chains. Subscriptions constructed via rclcpp::TypeAdapter
  template specialization — used by frameworks such as Isaac ROS
  NITROS — currently land their E vertex in the graph but skip the
  F-binding tracepoint. We measure this on the apriltag benchmark and
  observe 1 of 79 F vertices affected (1.3%); the underlying
  subscription still fires rcl_subscription_init and remains visible
  as an E. A targeted fish_nitros_compat_sub_cb_added tracepoint at
  the NITROS type-manager callback closes this gap for NITROS-based
  pipelines without modifying upstream rclcpp."

This honesty-up-front framing protects the framework from being
"caught" later and gives a clean path to fix when needed.

## What to do operationally

Short-term (next): add the NITROS-side patch (separate task), measure
zero-delta on apriltag, document in commit.

Medium-term: scan other Isaac ROS benchmarks for similar TypeAdapter-
boundary cases; add to high_impact/ if new patterns emerge.

Long-term: consider an upstream rclcpp PR adding the TypeAdapter
specialization tracepoint, so any TypeAdapter user gets attribution
for free.
