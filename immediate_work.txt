0. [URGENT] Batch process detection in GPU daemon
--------------------------------------------------
Problem:
  GPU daemon scans /proc one process at a time. Each new process triggers
  a settle_time wait (default 3s) before the next scan. With Autoware's
  74 processes, this means ~222 seconds (4 minutes) just for detection.

  Current flow (serial):
    detect PID 1234 → wait 3s → detect PID 1235 → wait 3s → ...

  Expected flow (batch):
    detect PID 1234, 1235, 1236, ... → wait 3s once → check all

Root cause:
  gpu.py daemon poll loop processes one new PID per cycle. Each cycle
  sleeps settle_time after detection. Should collect ALL new PIDs in
  one scan, then wait once, then process all.

Fix:
  In gpu.py daemon loop:
    1. Scan /proc for ALL new PIDs (not seen before) in one pass
    2. Wait settle_time ONCE
    3. Check all new PIDs for CUDA libraries in parallel
    4. Act on GPU processes (kill/resurrect for nsys)
    5. Mark all checked PIDs as seen

  This changes O(N * settle_time) → O(settle_time) for N processes.

  Also consider: reduce settle_time to 1.0s or 0.5s for non-GPU nodes
  (they don't need CUDA library loading time).

Discovered: Autoware has 74 processes, 4 minutes wasted on serial detection.

Priority: URGENT (blocks Autoware trace workflow)


1. [FIXED 2026-03-29] Add globalTid to nsys ingest (cuda_runtime table)
-----------------------------------------------------------------------
Problem:
  The nsys sqlite CUPTI_ACTIVITY_KIND_RUNTIME table contains a globalTid
  column that identifies which CPU thread made each CUDA API call.
  This column was NOT ingested into InfluxDB's cuda_runtime table.

  Current cuda_runtime columns:
    api_name, correlationId, duration_ns, eventClass, returnValue, source, time

  Missing: globalTid (thread identifier)

Why it matters:
  - In multi-threaded GPU nodes, different threads may issue CUDA API calls
    concurrently. Without globalTid we cannot distinguish which thread
    initiated which GPU operation.
  - For the current session (fish_20260322_212915) this is not blocking:
    yolo_inference_node has only 1 distinct globalTid in RUNTIME events.
  - For future sessions with multi-threaded CUDA nodes (e.g., TensorRT
    with multiple inference streams), this will be critical.

Fix:
  Update ingest.py to include globalTid when writing cuda_runtime points.
  Resolve globalTid to a local tid via the nsys sqlite StringIds or
  PROCESSES table if possible, or store the raw globalTid.

  Also consider adding globalPid to gpu_kernels and gpu_memcpy for
  multi-process nsys sessions (currently inferred from 'source' field).

NOTE: An initial attempt to add globalTid/globalPid was made but contains
  a bug — the fields are appended BEFORE `fields = []` initialization,
  so they are either lost or cause NameError. See known_bugs.txt BUG-002.
  This must be fixed before re-ingesting any session.

Priority: MEDIUM (not blocking current session, needed before next session)


2. [FIXED 2026-03-29] Fix nsys timestamp offset in ingest.py
-------------------------------------------------------------
Problem:
  All nsys-derived timestamps in InfluxDB are shifted ~54 minutes into
  the past. See known_bugs.txt BUG-003 for full analysis.

  Current code:  abs_ts = start + (utcEpochNs - systemClockNs)
  Correct:       abs_ts = utcEpochNs + start

  The "start" column in CUPTI activity tables is relative to the nsys
  session start, not a monotonic system clock value.

Open questions (investigate before fixing):
  - Is this behavior consistent across nsys versions? (tested: nsys 2024.x)
  - Does systemClockNs vs start semantics change for different CUPTI
    activity kinds (KERNEL vs RUNTIME vs MEMCPY)?
  - Would it be better to have model extraction read nsys sqlite directly
    instead of going through InfluxDB? (avoids the conversion entirely)
  - Need to test on a different system to confirm uptime is the offset source.

  If fixing ingest.py: also fix BUG-002 (globalTid/globalPid ordering)
  in the same change, since both are in _ingest_nsys_table().

  If the fix is confirmed correct, the current session's InfluxDB data
  should be re-ingested (drop and recreate the measurements).

Priority: HIGH (blocks temporal correlation between ros2_trace and nsys)


3. Trace-based edge filtering pre-process
------------------------------------------
Problem:
  Currently, parameter service filtering (SKIP_PARAMSERVICE) uses a static
  name blocklist. This works for /parameter_events but is not generalizable.

  A trace-based pre-filter would scan rcl_publish events BEFORE graph
  construction and compute per-(node, topic) publish counts and rates.
  Topics with negligible runtime activity would be excluded from edge
  construction, while topics that are actively used would be kept —
  even /parameter_events if a node genuinely uses it.

  See about_paramservice.txt for full analysis: in session fish_20260322_212915,
  /parameter_events accounts for 83/21496 publishes (0.4%), with 18 of 23
  nodes publishing exactly once (startup event only).

Implementation:
  Add a pre-filter step between model extraction and graph construction:
    1. Scan rcl_publish events, resolve publisher_handle -> topic
    2. Compute publish_count per (node, topic) pair
    3. Apply threshold (e.g., < 10 publishes → exclude)
    4. Return a filter-decisions dict for add_horizontal_relations()

  This replaces the static blocklist with evidence-based filtering and
  generalizes to any low-activity topic, not just parameter_events.

Priority: LOW (static filter is sufficient for now, needed when analyzing
  systems where parameter reconfiguration is part of the mission logic)


4. External source/sink detection and representation
-----------------------------------------------------
Problem:
  Some nodes subscribe to topics that no traced node publishes, or publish
  to topics that no traced node subscribes to. These topics cross the FISH
  trace boundary — their counterparts live in other containers, other
  machines, or non-traced processes (e.g., ros_gz_bridge, PX4 SITL).

  Currently these nodes appear "isolated" in the graph even though they
  are actively communicating — just not with anything inside the trace.

Observed in session fish_20260322_212915:

  state_sharing_node:
    Subscribes to (no publisher in graph):
      /Drone1/fmu/out/vehicle_global_position  (px4_msgs/msg/VehicleGlobalPosition)
      /Drone1/fmu/out/vehicle_local_position   (px4_msgs/msg/VehicleLocalPosition)
    Publishes to (no subscriber in graph):
      /state_sharing_drone_1                   (state_sharing/msg/SharedState)

    Sources are PX4 SITL / ros_gz_bridge (different container, not in trace).
    Destination is likely another drone or ground station (not in trace).

  px4_offboard:
    Subscribes to (no publisher in graph):
      /Drone1/fmu/out/airspeed_validated       (px4_msgs/msg/AirspeedValidated)
      /Drone1/fmu/out/vehicle_global_position  (px4_msgs/msg/VehicleGlobalPosition)
      /Drone1/fmu/out/vehicle_local_position   (px4_msgs/msg/VehicleLocalPosition)
      /Drone1/fmu/out/vehicle_odometry         (px4_msgs/msg/VehicleOdometry)
      /tracks                                  (ground_system_msgs/msg/SwarmObs)

    Same PX4/bridge source. /tracks comes from ground system (external).
    These are real dataflow inputs that are invisible in the current graph.

  transform_listener_impl_646f641f63a0:
    Internal tf2_ros node, auto-created when a node uses TransformListener.
    Most likely belongs to px4_offboard's process. Subscribes to /tf
    published by kiss_icp_node.

Impact:
  - Nodes with only external connections appear isolated (state_sharing_node)
  - Nodes with both internal and external connections (px4_offboard) show
    only partial connectivity — their real input set is larger
  - Graph metrics (degree, betweenness, dataflow roles) are skewed because
    external edges are missing

Implementation:
  1. After building sub_index and pub_index in add_horizontal_relations(),
     detect "dangling" topics:
       - Topics in sub_index with no match in pub_index → external source
       - Topics in node.A_v['publishers'] with no match in sub_index → external sink
  2. Create lightweight "external" placeholder vertices for these:
       - Type: "EXT" or use existing vertex types with an external flag
       - One vertex per external topic (or group by source system)
       - Add comm edges from/to these placeholders
  3. In visualization, render external vertices distinctly:
       - Gray dashed border (already done in detail views for "ext" references)
       - Labeled with topic name and direction (source/sink)
  4. Report a summary: which topics cross the trace boundary, which nodes
     are affected, and what percentage of a node's communication is external.

  The detail view (export_fish_detail) already creates gray "ext" reference
  nodes for cross-executor edges. Extend this pattern to true external
  sources/sinks at the graph construction level.

Priority: MEDIUM (affects graph completeness and metric accuracy, needed
  before publishing any dataflow analysis results)


5. Trace data volume vs accuracy tradeoff model
-------------------------------------------------
Problem:
  Current trace data volume is extremely large. A ~30 second simulation
  session produced:
    - ros2_trace: 8,628,771 events
    - gpu_kernels: 817,320 rows
    - cuda_runtime: 984,338 rows
    - nvtx_events: 214,208 rows
    - gpu_memcpy: 75,145 rows
    - Total ingest time: ~12 minutes

  This does not scale to longer sessions (minutes to hours). Both nsys
  and ros2_trace need an optimal data volume vs accuracy tradeoff.

  nsys side:
    nsys supports sampling frequency configuration. Lower frequency
    reduces data volume but may miss short-lived GPU kernels or bursty
    CUDA API patterns. We need a statistical model that, given a chosen
    sampling frequency, provides a confidence interval for the accuracy
    of derived metrics (kernel duration distributions, API call rates,
    memory bandwidth utilization).

    Approach:
      1. Use current full-rate trace as ground truth
      2. Subsample at various frequencies (1/2, 1/5, 1/10, 1/50, 1/100)
      3. For each subsample, compute key metrics (mean/p50/p99 kernel
         duration, API call rate, memory throughput)
      4. Compare against ground truth, derive error bounds
      5. Fit a model: f(sampling_frequency) → confidence_interval(metric)
      6. Recommend a default frequency that keeps error < X% at 95% CI

  ros2_trace side:
    LTTng captures every single event (publish, take, callback_start/end).
    For systems with high-frequency topics (e.g., IMU at 200Hz), this
    generates massive volumes. Options:
      - Filter by topic/event type at capture time (lttng enable-event)
      - Post-capture downsampling for rate-based analysis
      - Windowed capture (trace 5s every 30s) for steady-state profiling
    Need to quantify: how many seconds of full-rate trace are needed to
    accurately characterize callback latency distributions, scheduling
    jitter, and message flow rates?

  Deliverable:
    A script or notebook that takes a full-rate trace, performs the
    subsampling analysis, and outputs a recommendation table:
      | Sampling config | Data reduction | Metric accuracy (95% CI) |

Priority: HIGH (blocks scaling to real mission durations and multi-drone
  sessions — current 30s trace already takes 12min to ingest)


6. Automated simulation session with scripted actions
------------------------------------------------------
Problem:
  Current workflow is fully manual: start sim_run.sh, wait for nodes to
  come up, manually type action commands in the aircraft container's tmux
  pane, wait for completion, then stop FISH. This makes trace content
  unpredictable — we don't know exactly which actions were sent, when,
  or whether they completed.

  In session fish_20260329_182842, action execute events were lost due
  to LTTng buffer overflow (404 events discarded). We cannot tell if
  the actions would have been captured otherwise.

Why it matters:
  - Model validation requires known ground truth: which actions were
    sent, their expected service calls, and timing bounds
  - Reproducibility: re-running the same scenario should produce
    comparable trace data
  - CI/testing: automated trace capture enables regression testing of
    the FISH pipeline itself

Requirements:
  1. A mission script that sends a defined sequence of actions:
       - takeoff (wait for completion)
       - orbit or reposition (wait for completion)
       - land (wait for completion)
     Using the AAS CLI primitives:
       cancellable_action "ros2 action send_goal /Drone1/takeoff_action ..."
       cancellable_action "ros2 action send_goal /Drone1/orbit_action ..."
       cancellable_action "ros2 action send_goal /Drone1/land_action ..."

  2. The script should run INSIDE the aircraft container after nodes
     are up, either via docker exec or as a tmux pane command.

  3. Timing: wait for node initialization (e.g., 10s after container
     start), then execute the action sequence, then signal FISH stop.

  4. Output: a manifest file listing what was sent and when, so model
     extraction can cross-check against trace events.

  5. LTTng buffer size must be increased to avoid event loss. Add
     --subbuf-size-ust parameter to the ros2 trace invocation in
     install_fish.sh's trace_session.sh.

Priority: HIGH (blocks action tracepoint validation and reproducible
  trace sessions for paper results)


7. Speed up runtime aspect attribution
----------------------------------------
Problem:
  attribute_aspects() currently loads ALL callback_start/end + rcl_publish
  events from MongoDB per PID, merges into timeline, and scans. This is
  slow (~3 minutes for 9M events) because:
    - Multiple sequential MongoDB queries per PID
    - No compound index on (vpid, event, ts)
    - Full event loading into Python memory

  Two complementary approaches:

  A. MongoDB compound index (quick win):
     Add index on (vpid, event, ts) to ros2_trace collection.
     This would dramatically speed up the per-PID filtered queries.
     Implementation: add to ingest.py after collection creation:
       db["ros2_trace"].create_index([("vpid", 1), ("event", 1), ("ts", 1)])
     Risk: increases ingest time slightly, increases storage.

  B. Single-pass bulk load (larger improvement):
     Instead of querying per PID, load ALL relevant events once:
       - All callback_start/end events
       - All rcl_publish events
       - All rclcpp_client_request_sent events
     Group by vpid in memory, then process per-node.
     Risk: for very long traces, may exceed available memory.
     Mitigation: stream from babeltrace2 directly instead of MongoDB,
     or use windowed processing.

  Current timing: ~3 minutes for 9.1M event session (30s of trace data).
  Target: < 30 seconds.

Priority: MEDIUM (doesn't block functionality, blocks scaling to longer sessions)


8. Service client continuation function (split callback model)
---------------------------------------------------------------
Problem:
  When an entity has a client aspect (calls a service), its callback
  execution is split into two phases:

    Phase 1: entity callback runs, sends request (cli aspect)
    Phase 2: response arrives, handle_response runs (continuation)

  These are two separate executor dispatch cycles. Currently the model
  has a single function per entity — the response dep edge returns to
  the SAME function. This is incorrect: the continuation should be a
  separate function vertex.

  Correct model:
    [F] entity_cb ──request──> [F] server_cb
    [F] server_cb ──response──> [F] entity_cb_continuation

  Current model (wrong):
    [F] entity_cb ──request──> [F] server_cb
    [F] server_cb ──response──> [F] entity_cb  ← same function

Implementation:
  1. For each entity with cli aspect, create a second function vertex
     (continuation) linked to rclcpp_client_response_received trace event
  2. Response dep edge targets the continuation function, not the original
  3. Continuation function gets callback_start/end timing from
     handle_response (when callback variant is used)

  Requires: rclcpp_client_response_received tracepoint firing for
  internal node-to-node service calls (currently only fires for
  rclpy ActionClient CLI calls, not rclcpp internal clients).

  First real test case: Autoware on Jazzy — expected to have internal
  rclcpp node-to-node service calls where both client and server are
  persistent traced nodes.

  Dependencies:
    - Jazzy port (jazzy_port.txt)
    - Autoware trace session with internal service calls

Priority: HIGH (after Jazzy port — first thing to verify on Autoware run)


9. Per-callback roofline model via hardware performance counters
-----------------------------------------------------------------
Problem:
  FISH captures callback execution time (callback_start/end) but not
  computational characteristics. We don't know if a callback is
  compute-bound, memory-bound, or cache-contention-limited.

  A cache-aware roofline model per callback would enable:
    - Identify compute vs memory bound callbacks
    - Estimate shared bus contention between concurrent callbacks
    - Cache-related delay analysis (L1/L2/L3/DRAM hierarchy)
    - Data transaction volume per callback
    - Input for schedulability analysis (WCET decomposition)

Approach:
  Use Linux perf PMU counters system-wide alongside FISH tracing.
  PMU monitoring has a fixed, system-independent overhead (HW register
  reads, context switch save/restore). No per-process attach needed.

  Data collection (two concurrent streams):

    A. Per-CPU PMU counters:
       perf stat -a -A -e cycles,instructions,cache-misses,cache-references,\
         LLC-load-misses,LLC-store-misses,bus-cycles -I 1 \
         -o $SESSION_DIR/perf_counters.csv

       The -A flag reports per-CPU counters (not aggregated).
       Run concurrently with LTTng trace session. Start in trace_session.sh
       start_session(), stop in stop_session().

    B. CPU migration tracking:
       LTTng kernel tracepoint: sched_migrate_task
       Fires on every thread migration: (pid, old_cpu, new_cpu, timestamp)
       Much lower volume than sched_switch (only on migration, not every
       context switch).

       Requires: lttng-modules installed, root access, kernel tracing enabled.
       Add to fish_events.txt as kernel event:
         lttng enable-event -k sched_migrate_task

       Alternative (lower fidelity): use cpu_id field already present in
       every LTTng UST event — shows CPU at tracepoint fire time but misses
       inter-tracepoint migrations.

  Analysis (per callback, migration-aware):
    1. callback_start/end gives time window [t1, t2] and initial cpu_id
    2. sched_migrate_task events within [t1, t2] for this vtid give
       the CPU timeline: [(t1, cpu3), (t_mig, cpu7), ..., (t2, cpu7)]
    3. For each (cpu, time_slice) segment:
       - Read per-CPU PMU counter delta from perf_counters.csv
       - Attribute that delta to this callback
    4. Sum across all CPU segments → callback's total PMU values
    5. Derive per-callback metrics:
       - IPC = instructions / cycles
       - Cache miss rate = cache-misses / cache-references
       - LLC miss rate = LLC-load-misses / (LLC-load-misses + LLC-load-hits)
       - Operational intensity = instructions / (cache-misses * cache_line_size)
    6. Plot on cache-aware roofline

  Why migration tracking is critical:
    Without it, per-CPU counter attribution is incorrect:
      - Thread migrates cpu3→cpu7 mid-callback
      - cpu3's counter delta includes OTHER threads' work after migration
      - cpu7's counter delta includes OTHER threads' work before migration
      - Naive attribution double-counts or misattributes

    With sched_migrate_task:
      - Exact time slices per CPU known
      - Counter deltas split proportionally
      - Precise per-callback roofline datapoint

  GPU callbacks (via nsys):
    nsys already provides kernel-level compute and memory throughput.
    Can generate GPU roofline datapoints directly from:
      - Kernel duration + SM occupancy → achieved compute
      - memcpy bytes / duration → achieved bandwidth
      - Register usage → occupancy limit

  P-MoVE/SuperTwin integration:
    PMU abstraction layer from P-MoVE handles Intel/AMD/ARM event name
    differences. CARM model construction reuses P-MoVE's microbenchmark
    approach for establishing cache-level bandwidth ceilings.
    See reference_supertwin_pmove.md for details.

  Deliverable:
    - perf + sched_migrate_task integration in FISH trace session
    - Per-callback migration-aware PMU attribution
    - Roofline plot generation (cache-aware, separate CPU and GPU)
    - Reference: https://sips.inesc-id.pt/~ilic/roofline.php

Priority: HIGH (enables precise per-callback compute/memory characterization,
  shared bus contention analysis, and WCET decomposition — strong paper
  contribution building on SuperTwin/P-MoVE heritage)


10. Multi-container trace composition for unified task model
--------------------------------------------------------------
Problem:
  In multi-container deployments like AAS, each container runs a subset
  of the robotic system (aircraft, ground station, simulation). FISH
  currently traces a single container. The complete task model requires
  composing traces from multiple containers into a single graph.

  Example (AAS):
    - aircraft container: px4_interface, px4_offboard, yolo, kiss_icp
    - simulation container: ros_gz_bridge, PX4 SITL, Gazebo
    - ground container: ground station nodes

  Currently, topics crossing container boundaries appear as external
  entities (EXT). With multi-container composition, these EXT entities
  would be replaced by real entities from the other container's trace.

Approach:
  1. Install FISH in each container independently (setup_fish.sh)
  2. Run trace session in each container concurrently (synchronized
     via /clock topic or NTP)
  3. After collection, compose traces:
     a. Ingest each container's trace into separate MongoDB databases
     b. Extract model per container (identify_executors → entities → functions)
     c. Merge models: match EXT entities across containers by topic/service name
        - EXT publisher in container A matches real subscriber in container B
        - EXT subscriber in container A matches real publisher in container B
        - Replace EXT→real edges with cross-container real→real edges
     d. Unified graph: all executors from all containers, cross-container
        edges marked with container origin and network latency

  Timestamp alignment:
    - All containers share /clock topic (simulated time) — timestamps align
    - For real deployment: NTP sync + clock offset correction
    - Cross-container message latency measurable from publish→take timestamps

  Visualization:
    - D3 interactive: color-code nodes by container origin
    - Grafana: per-container dashboards + unified cross-container view

  Deliverable:
    - fish_compose.py: takes N session directories, merges into unified model
    - Cross-container edge detection via topic/service name matching
    - Unified fish_graph.json for D3 visualization

Priority: HIGH (required for complete system model in multi-container
  deployments — paper contribution: "FISH composes distributed traces
  into a unified scheduling hierarchy")
